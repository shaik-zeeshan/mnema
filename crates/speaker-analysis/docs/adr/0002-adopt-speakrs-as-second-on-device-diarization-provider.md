# Adopt speakrs as a Second On-Device Diarization Provider, Trending to Replacement

> **Amended by [ADR 0003](0003-remove-sherpa-make-speakrs-sole-diarization-provider.md).** The "eventual replacement" and "default flip" this ADR anticipated have happened: sherpa-onnx is removed and `speakrs` is now the **sole** on-device diarization provider and the default — no longer a second provider beside sherpa. The provider-adoption rationale and benchmark evidence below stand; the coexistence/opt-in framing is superseded.

We add **speakrs** (pure-Rust pyannote `community-1` segmentation + WeSpeaker embedding + VBx clustering, native CoreML) as a second **Speaker Analysis** provider beside the on-device sherpa-onnx runtime, adopted on a measured **accuracy + dependency-simplification** basis: it beat the production baseline on the full 232-clip VoxConverse split (8.35% vs ~10.9% DER), degrades gracefully across speaker counts, and drops the native sherpa-onnx/NeMo dependency. It ships as exactly **one non-default Speaker Model Preset** (sherpa stays the default) and is designed for **eventual replacement** of sherpa, not permanent coexistence — Mnema's release target is Apple Silicon macOS only, so speakrs's CoreML constraint excludes no shipping platform. It runs **whole-segment** through the existing subprocess-per-job helper, its WeSpeaker output forms a new **Voiceprint Space**, and its models load from the existing speaker-analysis model store.

## Considered Options

- **Stay on sherpa-onnx** — rejected. It chronically over-clusters (mean speaker-count error +17.9) and keeps a native sherpa-onnx/NeMo dependency with documented failure modes (30s pyannote SIGBUS, TitaNet aborting past ~60s of embedding audio). speakrs beat it at full scale and holds a tight 4.6–9.6% DER band across every speaker count.
- **sortformer (NVIDIA Streaming Sortformer v2.1) as the default instead** — rejected. Sharper at ≤2 speakers, but a hard 4-speaker architectural cap makes it explode above it (28.78% DER at 8+ speakers vs speakrs's 9.56%). Parked as a possible future low-count / live-streaming path only if capture telemetry skews ≤2 speakers; it is not this decision.
- **Chunk speakrs to a 60s safe-window like sherpa** — rejected. speakrs clusters *globally* via VBx and was measured whole-clip; chunking would force cross-chunk re-stitching and reintroduce the over-clustering the benchmark avoided — shipping an algorithm we never measured. The 5-minute **Audio Segment** cap (ADR 0005) plus subprocess-exit memory reclamation already bound the transient CoreML peak (~2.3 GB on a worst-case segment) without chunking. Safe-chunking stays specific to sherpa's own per-call ceilings.
- **Surface speakrs as a user-facing "engine" selector (two-level engine → preset)** — rejected. The domain's user-facing unit is the **Speaker Model Preset**; `provider` is an internal descriptor attribute. A prominent engine picker is UX we would tear out on the replacement path. speakrs is one more preset; the provider axis stays internal.
- **Per-instance provider identity ([ADR 0035](../../../../docs/adr/0035-provider-identity-is-a-per-instance-id-not-the-kind.md))** — not applicable. That governs user-connected AI endpoints with keychain secrets, where two of one kind coexist. Diarization engines are a fixed, built-in set, so `provider` stays a fixed *kind* string (`sherpa_onnx`, `speakrs`), like the OCR and audio-transcription provider enums.
- **speakrs's own `from_pretrained` HF auto-download, or bundling the ~321 MB** — rejected. Auto-download bypasses the preset model-management surface (no size disclosure, progress, SHA256 verification, marker files, or retention visibility — violating ADR 0001's "download size shown up front"), landing weights in an opaque cache outside the store. Bundling bloats an opt-in preset's installer and matches nothing else (sherpa/whisper/parakeet all download). Models go through the existing descriptor/download infra and load via `from_dir` / `SPEAKRS_MODELS_DIR`.
- **A statistical real-capture A/B before flipping the default** — rejected as infeasible. Real Mnema captures carry no ground-truth speaker labels, so DER is uncomputable there; a labeled-cohort harness would be theater. The default-flip is an explicit judgment call.

## Consequences

- speakrs is **opt-in until a judgment-call flip**: the VoxConverse win re-confirmed through the bench harness's `--binary` path (owned data, not new data), plus operational no-regression on real captures (job-failure rate, wall-clock under the 600s helper timeout, subprocess memory), plus dogfood quality at least as good as sherpa. To make the over/under-clustering proxy observable, **clusters-per-segment and analysis duration are added to successful-job result provenance**. No statistical A/B harness is built.
- Selecting speakrs creates a new WeSpeaker **Voiceprint Space** (256-d), not comparable to sherpa's TitaNet space. Enrollments persist per `model_id` and go **dormant**, not deleted, on a switch (per ADR 0001). Eventual sherpa *removal* makes the TitaNet voiceprints permanently unreachable — so removal must warn enrolled users and must not land until speakrs has been the default long enough for organic re-tagging.
- speakrs ships as **exactly one preset**; on the replacement path the preset list shrinks toward a single entry. The preset machinery (per-`model_id` keying, **Speaker Continuity**, the download surface) still applies with one entry, and validated sub-presets can be added later — we do not invent unvalidated knob-presets to backfill the sherpa Balanced/Multilingual/High-accuracy trio speakrs was not built for.
- The subprocess-per-job helper **generalizes beyond sherpa**: it dispatches on the request's `provider`, and the subprocess-exit boundary (already the memory-reclamation boundary) is what makes whole-segment speakrs safe. A `speakrs` Cargo feature gates the dependency at compile time (on in release); it is not the user-facing switch.
- **`recognize_saved_people` works out of the box.** speakrs's public `DiarizationResult` exposes per-chunk WeSpeaker embeddings keyed to the same cluster ids as its segments; per-speaker centroids are mean-pooled into `SpeakerCluster` (~40 lines of ndarray). No fork, no separate embedder, no model-space mixing.
- Two follow-ups are tracked **separately**: (1) enumerate and verify the `avencera/speakrs-models` artifacts (filenames, sizes, SHA256, layout) and the governing license from the HF model card, feeding the manifest descriptor and its `license_label`; (2) build an app-wide third-party attribution surface — a pre-existing gap (sherpa's pyannote/NeMo are already unsurfaced) that speakrs's attribution license forces into the light, closed as its own compliance task rather than a speakrs-only band-aid.
- The "faster / lower power" goal parked in ADR 0001 remains **formally unmet**: speakrs is CoreML and likely helps, but this round measured accuracy and memory, not power, so no speed/power claim is made.
