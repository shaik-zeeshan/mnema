# Remove sherpa, Make speakrs the Sole On-Device Diarization Provider

We remove the **sherpa-onnx** diarization provider entirely and make **speakrs** (pure-Rust pyannote-community-1 segmentation + WeSpeaker embedding + VBx clustering on native CoreML, model id `pyannote-community-1-wespeaker`) the **only** on-device **Speaker Analysis** provider and the default. The two-provider coexistence and "eventual replacement" path that [ADR 0002](0002-adopt-speakrs-as-second-on-device-diarization-provider.md) set up is now realized: speakrs is no longer a second, opt-in preset beside sherpa — it is the single shipping preset, and the sherpa-onnx/NeMo native dependency, its presets, and its TitaNet **Voiceprint Space** are gone. This **supersedes [ADR 0001](0001-speaker-model-presets-and-voiceprint-scope.md)** (whose Balanced/Multilingual/High-accuracy sherpa-CPU presets no longer exist) and **amends [ADR 0002](0002-adopt-speakrs-as-second-on-device-diarization-provider.md)** (whose default-flip judgment call is now taken).

## Considered Options

- **Keep sherpa as a fallback/secondary provider** — rejected. Carrying a removed provider's native dependency, model presets, download surface, and dispatch arm purely as a fallback keeps the documented sherpa-onnx/NeMo failure modes (30s pyannote SIGBUS, TitaNet aborting past ~60s of embedding audio) and the chronic over-clustering (mean speaker-count error +17.9) alive for no accuracy benefit. Mnema's release target is Apple Silicon macOS only, so speakrs's CoreML constraint excludes no shipping platform, and a second provider would only be reachable through UX (a provider/engine selector) that ADR 0002 already decided not to surface.
- **Keep both providers until a Windows/Linux port needs a non-CoreML engine** — rejected for now. The Windows port is unscheduled research (`docs/windows/media-processing-research.md`), and that research can pick a cross-platform diarization engine when the port is real. Holding a whole second provider in the shipping macOS build against a hypothetical future port is the wrong place to pay that cost; the port will re-introduce a non-CoreML provider deliberately, not inherit sherpa by inertia.
- **Drop the `provider` axis now that there is one provider** — rejected. Keeping `provider` as an internal descriptor attribute (now effectively `speakrs`) is cheap, preserves the dispatch seam where a future Windows/ANE provider would slot in, and is what the legacy-settings remap keys off. Collapsing it would make re-introducing a provider a larger change for no present gain.

## Migration

- **Legacy `sherpa_onnx` settings and jobs are remapped to speakrs at the validation/dispatch seam.** A user (or an in-flight/queued **Speaker Analysis Job**) carrying provider `sherpa_onnx` or one of the old sherpa `model_id`s is remapped to the speakrs preset rather than failing — there is no sherpa dispatch arm to route to. Existing recordings keep their original-preset results; only new and reprocessed analysis runs through speakrs.
- **TitaNet/CAM++ enrollments become unreachable.** sherpa's voiceprints lived in a TitaNet (or CAM++) **Voiceprint Space** keyed to the removed `model_id`s; with the provider gone, those enrollments can no longer be matched. Per ADR 0001/0002 enrollments are not deleted on a preset change, but sherpa *removal* makes them permanently dormant. Recognition re-grows organically as people are re-tagged under the speakrs WeSpeaker (256-d) space.

## Consequences

- **speakrs is the default and only diarization provider.** Its compile-time feature (`speaker-analysis-speakrs`) is on by default in `apps/desktop/src-tauri`; the default desktop build therefore links OpenBLAS, built from source and statically linked via speakrs' `openblas-static` feature so the shipped binary has no `libopenblas.dylib` runtime dependency (see the repo `AGENTS.md`/`CLAUDE.md` verification note — `brew install gcc` at build time, force-static Fortran runtime in the desktop `build.rs`). The `speaker-analysis-sherpa-onnx` feature and the native sherpa-onnx/NeMo dependency are removed.
- **The multilingual and high-accuracy presets are dropped; only the speakrs preset remains.** speakrs was not built to backfill the sherpa Balanced/Multilingual/High-accuracy trio, and ADR 0002 already refused to invent unvalidated knob-presets to do so. The **Speaker Model Preset** machinery (per-`model_id` keying, **Speaker Continuity**, the download surface) still applies with a single entry, and validated presets can be added later.
- **The accuracy and dependency case is settled by the ADR 0002 benchmarks.** speakrs beat the sherpa production baseline on the full 232-clip VoxConverse split (~7.4–8.4% DER vs sherpa's ~10.9%), holds a tight band across every speaker count (degrading gracefully where sherpa over-clusters), and drops the native sherpa-onnx/NeMo dependency for a pure-Rust + CoreML pipeline. Because real Mnema captures carry no ground-truth speaker labels, no new statistical A/B is run; the removal is the explicit judgment call ADR 0002 reserved.
- **speakrs safe-chunks long segments (revised — see amendment).** This ADR originally shipped speakrs whole-segment with no safe-chunk window, reasoning that the transient CoreML peak was bounded by the 5-minute **Audio Segment** cap and reclaimed at the subprocess-exit boundary. That reasoning was an assumption; measurement disproved it (see the amendment). speakrs now safe-chunks segments longer than `SPEAKRS_SAFE_CHUNK_SECONDS` (180 s) and runs shorter ones whole. The generic dispatch `--safe-chunk-ms` flag remains accepted-and-ignored — speakrs's window is a fixed internal constant, not a tunable.
- **Third-party attribution narrows to speakrs.** The app-wide attribution surface tracked as a follow-up in ADR 0002 now needs to list only the speakrs pyannote-community-1 + WeSpeaker artifacts and their license (CC-BY-4.0), not sherpa's pyannote/NeMo lineage.

## Amendment (2026-06): safe-chunking restored, with cross-chunk stitching

The "no chunking — the 5-minute cap bounds the peak, which only frees on process exit" rationale was never measured. A footprint timeline + length sweep on real multi-speaker audio showed the whole-segment 5-minute peak is a **~5 GB upfront transient** with a discrete jump past **~3 minutes**, and that it frees *mid-run* (steady state ~2 GB; resident only ~1.3 GB) — it is not retained graphics memory waiting on the subprocess to exit. The CoreML compute-unit choice (GPU `All` vs CPU+ANE) does **not** move the peak, so that lever was dropped.

Diarizing the segment in **≤180 s chunks through the same pipeline** (CoreML sessions stay loaded) caps the peak at **~1.45 GB (3.6×)** and is *faster*. Because each chunk clusters independently, per-chunk speaker clusters are stitched back into segment-wide identities by **centroid cosine similarity**; benchmarking on the VoxConverse subset found a threshold of **0.6** is DER-neutral (**7.56 % vs 7.47 %** whole-segment), whereas 0.5 over-merges (+3.4 pp confusion) and 0.8 over-splits. A max-length 5-minute segment becomes two chunks (one stitch boundary); default-length segments stay under the window and run whole, so behavior is unchanged for the common case.

Implementation: `providers/speakrs.rs` (chunk loop + `SPEAKRS_SAFE_CHUNK_SECONDS` / `SPEAKRS_STITCH_SIMILARITY` constants) and `providers/speakrs_mapping.rs` (`stitch_chunk_mappings`, unit-tested). Provenance gains `chunkingMode = "safe_chunked"`, `chunkCount`, and `safeChunkSeconds` when a segment is chunked.
