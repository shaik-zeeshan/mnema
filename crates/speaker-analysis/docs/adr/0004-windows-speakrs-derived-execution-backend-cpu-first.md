# Windows speaker-analysis runs speakrs on a derived execution backend; CPU ships first, CUDA deferred

Windows gains on-device diarization by running the existing speakrs **Speaker Model Preset** (`pyannote-community-1-wespeaker`) on a derived **Execution Backend**, not a new provider or model. v1 ships the **CPU** backend with statically-linked ONNX Runtime; CUDA is a deliberate follow-up. Backend is orthogonal to identity — it never changes `model_id`, **Voiceprint Space**, or **Speaker Continuity** keying, is chosen automatically at execution time, and is observable only in provenance (`executionMode`).

## Considered Options

- **CPU and GPU as two model-list entries (two Speaker Model Presets)** — rejected. `model_id` is the identity authority for the **Voiceprint Space** (`person_voice_embeddings.model_id`) and **Speaker Continuity** keying (`session_id, provider, model`). CPU and CUDA run the *same* WeSpeaker embedding model, so two presets would fork one comparable voiceprint space and break continuity across a pure-hardware distinction. It also makes the desired try-CUDA-fall-back-to-CPU behavior cross identity mid-job — the two-presets framing and the fallback are mutually incompatible.
- **A user-facing Acceleration setting (Auto/CPU/GPU)** — rejected for now. Backend is hardware, not user intent; ADR 0001 chose "intent, not knobs," and the crate already treats execution-time hardware choices as internal (the safe-chunk window is "a fixed internal constant, not a tunable"; the CoreML compute-unit lever was measured and dropped). A future "Force CPU" override could be added as a pure settings addition without touching identity.
- **Ship CPU + CUDA together** — deferred. CUDA carries the heavy, EULA-laden distribution problem (cu12 provider DLLs, CUDA 12 / cuDNN 9, GB-scale footprint). CPU-first restores parity — Windows has no diarization today — and mirrors the audio-transcription context's "GPU execution providers (DirectML, CUDA) are a future, separately-decided change." ADR 0003 anticipated this seam: "the port will re-introduce a non-CoreML provider deliberately."
- **`load-dynamic` linking now** — rejected for v1. `speakrs` inherits `ort`'s `download-binaries`, so a CPU-only Windows build static-links ONNX Runtime and ships zero separate DLL. Because `ort` features unify across the one `mnema.exe`, enabling `ort/load-dynamic` would flip parakeet (the other ONNX consumer in the same binary) to dynamic loading too — pure overhead for no CPU-path benefit. `load-dynamic` is what CUDA's dynamic provider DLLs actually need, so it is owned by the CUDA follow-up.

## Consequences

- One platform-stable `model_id`; only the on-disk artifact is platform-conditional (CoreML `.mlmodelc` set on macOS, `.onnx` + `.npy` subset on Windows). A voiceprint enrolled on one backend stays comparable to one computed on another.
- Windows audio decode goes through `media-decode` / Media Foundation (ADR 0024) to mono f32 @ 16 kHz, feeding the unchanged platform-agnostic `pipeline.run(&samples)`.
- Windows scope is x86_64 (`intel-mkl-static` via the shared `default-linalg` feature; CUDA is x86_64-only). Windows ARM64 is out of scope. Diarization is on-by-default on Windows, mirroring macOS.
- The CUDA follow-up owns, as a separately-decided unit: the `load-dynamic` switch (which also moves parakeet's Windows ORT to dynamic loading), bundling the redistributable ORT provider DLLs (`onnxruntime_providers_cuda/shared.dll`), runtime CUDA detection, and the distribution model. The distribution model is deferred; two constraints hold regardless — the base installer never bundles CUDA/cuDNN (NVIDIA EULA + size), and the shortlist is a downloadable GPU pack vs. requiring a user-installed CUDA toolkit located via `ort`'s `preload_dylibs`.
- A CUDA-init failure that falls back to CPU is a *successful* **Speaker Analysis Job** (it ran on CPU), not a runtime failure.
