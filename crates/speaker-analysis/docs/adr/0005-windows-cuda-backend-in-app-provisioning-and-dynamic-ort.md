# Windows CUDA Execution Backend: in-app NVIDIA-redist provisioning, Windows-scoped dynamic ORT, init-only fallback

The CUDA **Execution Backend** — ADR 0004's deferred follow-up (#137) — activates on Windows x86_64 by running the existing `pyannote-community-1-wespeaker` **Speaker Model Preset** through speakrs's `ort` CUDA execution provider. It is gated behind an **in-app, opt-in GPU Acceleration Pack** that downloads the NVIDIA CUDA 12 + cuDNN 9 *redistributables* into the app data dir (loaded via `ort`'s `preload_dylibs`); the base installer never ships, and we never host, any NVIDIA library. The backend stays orthogonal to identity: one `model_id`, auto-selected at execution time, observable only in provenance (`executionMode`).

## Decided

- **Distribution is in-app, on-demand, opt-in provisioning — not bundling, not an installer step.** The running app detects an NVIDIA GPU (NVML / `nvcuda` probe); the first time a GPU-eligible **Speaker Analysis Job** appears it offers to download the GPU pack into the app data dir, reusing the existing on-demand model-download machinery. It is a *separate* step from the model download — the model is identity (Voiceprint Space, Speaker Continuity), the pack is hardware.
- **We provision the NVIDIA redistributables ourselves; no system CUDA toolkit is required.** The pack fetches the redistributable subset (CUDA 12 runtime — `cudart64_12.dll`, `cublasLt64_12.dll`, … — and cuDNN 9) from NVIDIA's official endpoints into the app dir, loaded at runtime via `preload_dylibs` (no admin, no `PATH` changes, no global install). We are an orchestrator, not a redistributor: NVIDIA remains the distributor and the user accepts NVIDIA's license in-app.
- **All MIT ORT DLLs ship in the base install, version-locked together** — `onnxruntime.dll`, `onnxruntime_providers_shared.dll`, `onnxruntime_providers_cuda.dll`. They must match the pinned `ort` / `ort-sys` build exactly, so they travel with the binary as one unit. The GPU pack is then 100% NVIDIA-from-NVIDIA, with nothing of ours to version-skew.
- **`load-dynamic` is enabled binary-wide but scoped to the Windows target.** CUDA's provider DLLs require `ort/load-dynamic`, which is one feature on the single shared `ort` node — so it also moves parakeet's Windows ORT to dynamic loading. It is added only under `[target.'cfg(windows)'.dependencies]`, leaving the macOS build static.
- **v1 backend is `ExecutionMode::Cuda` (1-second step) only.** `CudaFast` / `coreml-fast` are out of scope (see below).
- **Observable fallback, init-failure only.** `executionMode` records the backend that *actually* ran. On a CUDA-init fallback it is `"cpu"`, plus `executionModeRequested = "cuda"` and `cudaFallbackReason = "<error>"`. A CUDA failure *after* successful init is a job failure, not a silent CPU re-run. The helper only *attempts* CUDA — and only records a fallback — when the pack is present; a GPU machine with no pack is plain CPU with no fallback noise (a *not-provisioned* state is not a failure).
- **A Windows-only "Use GPU acceleration" override ships in v1.** Settings exposes the GPU Acceleration *provisioning state* — detect GPU → offer pack → installed/working/failing, surfacing `cudaFallbackReason` in the UI — plus a default-on toggle that caps backend selection at CPU when off. This is the Force-CPU override ADR 0004 anticipated, pulled into v1: identity-safe (CPU and CUDA share `model_id` / Voiceprint Space / Continuity keying), read live at execution time (not frozen at admission), Windows-only (macOS always runs CoreML, no pack, no toggle). Settings exposes *provisioning + this single override* and is **never** a model/backend selector — there is no GPU model-list entry.

## Considered options

- **Bundle NVIDIA CUDA/cuDNN in the base installer** — rejected. NVIDIA EULA + GB-scale footprint; the ADR 0004 constraint that the base installer never bundles CUDA/cuDNN.
- **Installer-time provisioning** (NSIS/MSI detects the GPU and downloads the pack during setup) — rejected. Requires custom download logic in *two* installer formats, blocks/extends setup, breaks behind corporate proxies, and is redundant with the in-app path that fires moments later with better failure handling (network failure → app still runs on CPU, vs. → setup limps). The product is consumer / `currentUser`, not an enterprise silent-deploy story where install-time provisioning would earn its keep.
- **Require a user-installed CUDA 12 / cuDNN 9 toolkit, located via `preload_dylibs`** — rejected. Even a high-end NVIDIA dev box has no CUDA toolkit or cuDNN by default (verified locally: RTX 5070 Ti + driver present, `nvcuda.dll`/`nvml.dll` present, but no `cudart64_12.dll` and no cuDNN 9). Requiring a system install would strand most GPU owners. We provision the redist subset into our own dir instead.
- **A separate diarization helper *binary* with its own dynamic `ort`, leaving parakeet's ORT static** (to contain the `load-dynamic` blast radius) — rejected. It buys only "don't touch parakeet," at the cost of a new build target and a second `ort-sys` compile (~2× ORT footprint); and the crash-isolation it would justify is already provided by the speaker-analysis helper being a subprocess re-invocation of `mnema.exe`.
- **`CudaFast` / `coreml-fast` (2-second step)** — rejected for v1. The `-fast` cadence is a precision/speed quality trade-off (speaker-turn boundaries land coarser), not a hardware path. Auto-selecting it would make the GPU silently produce coarser results than the CPU — violating the "orthogonal to identity" Execution Backend invariant; exposing it as a user choice would reintroduce the Acceleration knob ADR 0004 rejected ("intent, not knobs"). Plain `Cuda` is already 50–121× realtime, so the `-fast` speedup buys little for a background job. A deliberate precision/speed axis, if ever wanted, is a separate decision that would span all backends.

## Consequences

- The Windows base build **loses its zero-ORT-DLL property**: it now ships three MIT ORT DLLs and loads ONNX Runtime dynamically. **parakeet's Windows ORT also goes dynamic** (the shared `ort` node) and must be re-verified on Windows. macOS is untouched — `load-dynamic` / `cuda` are Windows-target-only `ort` features, and macOS speakrs is native CoreML (no `ort`), so only macOS parakeet uses `ort` and it stays static via `download-binaries`.
- `derived_execution_mode()` stops being a pure compile-time `cfg` constant on Windows. It becomes runtime detection — NVML probe → try `Cuda` (with CPU fallback) → else `Cpu` — and the `executionMode` provenance reflects the runtime outcome rather than a compile-time string.
- The CUDA→CPU fallback is *ours* to drive: speakrs's CUDA arm uses `.error_on_failure()`, so a CUDA-init failure surfaces as an `Err` from `OwnedDiarizationPipeline::from_dir(.., Cuda)`; we catch it and re-run with `Cpu`. The fallback run is a **successful Speaker Analysis Job** (`executionMode = "cpu"` + the `executionModeRequested` / `cudaFallbackReason` diagnostics), surfacing no user-facing error.
- A machine with no NVIDIA driver runs on CPU with no pack offered and no error. A machine with a GPU but no pack installed runs on plain CPU with **no** fallback diagnostics (CUDA is never attempted — *not-provisioned* is not a failure); the Settings pack offer is what signals the opportunity. A fallback diagnostic (`cudaFallbackReason`) appears only when the pack *is* present and CUDA still fails to initialize.
- The pack contents are version-coupled to the pinned `ort` build (CUDA 12.x ↔ cuDNN 9.x; cuDNN 8.x is incompatible). The exact redist versions are pinned alongside the `ort` pin and bumped deliberately.
- The live CUDA path (and its CPU fallback) cannot be CI'd — it needs a real NVIDIA GPU plus the redist pack. The on-device pass is an operator checklist: [Windows CUDA Diarization — On-Device Verification Matrix](../windows-cuda-verification.md) (run on the RTX 5070 Ti fixture, which has GPU + driver + `nvml.dll` but no system CUDA/cuDNN — the native fallback fixture).
