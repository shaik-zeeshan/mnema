# AGENTS

## Workspace
- This repo has two roots of truth: the Bun/Turbo workspace in `package.json` and the Rust Cargo workspace in `Cargo.toml`.
- `apps/desktop` is the only JS app. Shared native/backend code lives in `crates/*`, not `packages/*`.
- Platform-specific support status and implementation checklists live in `SUPPORTS.md`; update it when adding or changing macOS, Windows, or Linux support behavior.

## Commands
- Run repo commands from the repo root.
- Frontend flow: `bun run dev`, `bun run check`, `bun run build`.
- Tauri CLI from the root: `bun run tauri -- dev` or `bun run tauri -- build`.
- Rust-only verification for the desktop app: `cargo check --manifest-path apps/desktop/src-tauri/Cargo.toml`. This requires the `mnema-cli` sidecar to exist first (see Verification) — a clean checkout will not have it.
- Focused Rust work: `cargo check -p <crate>` and `cargo test -p <crate>`.

## Boundaries
- Svelte UI lives in `apps/desktop/src`. The shell is `apps/desktop/src/routes/+layout.svelte`, the dashboard is `apps/desktop/src/routes/+page.svelte`, and settings live in `apps/desktop/src/routes/settings/+page.svelte`.
- Tauri startup and command registration live in `apps/desktop/src-tauri/src/lib.rs`; keep new `#[tauri::command]` functions wired there and keep frontend `invoke(...)` names in sync.
- The macOS status-bar/tray menu is Rust-owned in `apps/desktop/src-tauri/src/status_bar.rs`; keep it as a native Tauri `TrayIconBuilder`/`Menu`/`CheckMenuItem` surface rather than adding a Svelte tray UI or AppKit/Swift layer. Its stable menu IDs use the `tray_*` prefix, and the template icon asset is `apps/desktop/src-tauri/icons/tray-template.png`.
- The owning seam for native capture is `apps/desktop/src-tauri/src/native_capture/lifecycle.rs` (**Recording Lifecycle**); Tauri command handlers and background hooks should stay thin adapters over that module rather than orchestrating `NativeCaptureRuntime` directly.
- `crates/app-infra` owns SQLite, embedded sqlx migrations, background jobs, frame batches, and the OCR/processing pipeline.
- `crates/capture-types` owns serde types shared across frontend, Tauri, and native layers.
- `crates/capture-screen`, `crates/capture-microphone`, and `crates/capture-writers` own capture primitives and media output.
- `crates/capture-runtime` holds generic runtime helpers.

## Quirks
- The desktop frontend is SPA-only: `apps/desktop/src/routes/+layout.ts` sets `ssr = false`, and `apps/desktop/svelte.config.js` uses `@sveltejs/adapter-static` with `fallback: "index.html"`.
- In production desktop packaging, the SPA entry can arrive as `"/index.html"` instead of `"/"`; route-gating logic in the shell should normalize static-entry paths before deciding whether a surface is the main route.
- Tauri expects Vite on port `1420` with HMR on `1421`; `apps/desktop/vite.config.js` hard-pins those ports and ignores `src-tauri/**` in the watcher.
- The Tauri bundle identifier is `com.shaikzeeshan.mnema`; changing it changes the app identity used for Tauri/macOS config and cache directories.
- Recording settings persist to `recording-settings.json` under Tauri `app_config_dir()` when available. Keyboard shortcut preferences persist separately to `keyboard-bindings.json` under Tauri `app_config_dir()` and are Rust-owned in `apps/desktop/src-tauri/src/keyboard_bindings.rs`; global shortcut registration should stay there rather than in Svelte. App infra state lives directly under `<saveDirectory>`, with SQLite at `<saveDirectory>/db/app.sqlite3`; changing `saveDirectory` changes that DB location on the next app start.
- Dismissible one-time app prompts persist through shared One-Time Prompt State under Tauri `app_config_dir()`, not in `recording-settings.json`, browser local storage, or prompt-specific files. Prompt IDs should be stable/versioned and track shown/dismissed/completed timestamps.
- Native capture output is date-organized under `<saveDirectory>/recordings/YYYY/MM/DD/`. Screen recordings are saved as `<session>-segment-####.mov`, while audio stays separate under `<saveDirectory>/recordings/YYYY/MM/DD/audio/<session>/segment-####/`. Hidden per-segment workspace directories under the same date folder hold temporary screen capture artifacts and exported JPEG frames.
- Hidden segment workspace cleanup safety depends on the sibling visible segment `<session>-segment-####.mov`: if that file is missing or unopenable, app-infra preserves non-empty hidden workspaces for fallback/debugging, but empty/no-reference hidden workspaces are safe to remove. Existing stale safe workspaces can be repaired headlessly with `cargo run -p app-infra --bin repair_hidden_segment_workspaces -- <saveDirectory>`, and the desktop Tauri app also runs this repair once at startup plus every 5 minutes in the background.
- Hidden segment workspace repair now treats an existing-but-unopenable sibling segment `.mov` (for example, a file missing `moov`) the same as a missing visible segment for cleanup safety, so broken visible segments preserve hidden frame fallbacks instead of deleting the only preview source.
- Screen frame preview lookup now prefers a sibling per-segment binary sidecar `<session>-segment-####.frame-index.bin` written by `crates/capture-screen` during screen output finalization. The sidecar stores only captured-at/frame-index identity plus finalized-video-relative offsets derived from the finished asset's sample timing, and `apps/desktop/src-tauri/src/app_infra.rs` falls back to the older first-frame timestamp estimate only when neither the binary sidecar nor a legacy JSON sidecar is present.
- `get_frame_preview` now returns asset-backed preview file paths instead of base64 payloads. Stable persisted frame images are exposed directly through Tauri asset scope, while unstable fallback/video previews are materialized into app-owned cache files under Tauri `app_cache_dir()/frame-previews/` so hidden segment workspace cleanup/finalization cannot invalidate the frontend preview path.
- Scrub Preview is a separate low-cost dashboard navigation tier, not a source of **Captured Frame** truth. Generated Scrub Previews may be lower-resolution/timing-tolerant placeholders during timeline movement, but OCR/copy/download and parked exact inspection must continue through `get_frame_preview`; video-backed Scrub Preview generation should require indexed finalized segment timing rather than guessed offsets.
- Generated Scrub Preview artifacts live under Tauri `app_cache_dir()`, keyed by finalized screen segment/video-offset interval/rendition/source freshness, not under `<saveDirectory>/recordings` and not by Captured Frame id.
- Generated preview cache files under Tauri `app_cache_dir()/frame-previews/` are disposable and app-owned: startup and on-demand preview writes opportunistically clean them by age and count, and backend in-memory preview cache entries are invalidated when their cached preview file path no longer exists on disk.
- Existing legacy JSON sidecars can be converted headlessly with `cargo run -p app-infra --bin convert_frame_index_sidecars -- <saveDirectory>/recordings`.
- Segment-based retention is modeled in `capture_sessions` and `capture_segments`; retention applies only to the current `saveDirectory` / active DB, uses local-calendar cutoffs, preserves `person_profiles`, and should prune dashboard data through `timeline_data_changed` instead of forcing a timeline reset.
- Capture Segment Duration is capped at 5 minutes. This caps each rotated `Capture Segment`, not total `Capture Session` length, and bounds finalized segment work such as frame-index handling and automatic Scrub Preview Generation.
- App infra schema changes belong in `crates/app-infra/migrations`; migrations are embedded via `sqlx::migrate!`.
- `FrameBatchStore` has transaction-scoped finalize helpers for batched frame insertion; keep finalize-job scheduling inside the same transaction as frame persistence, OCR job planning/enqueue, and batch attachment when changing `insert_frame_into_batch_and_maybe_enqueue_ocr_job` or related paths.
- OCR dedupe is two-stage: `crates/capture-screen` computes captured-frame equivalence data, and `crates/app-infra` skips a new OCR job when an earlier equivalent frame in the same session is already eligible as the OCR fallback.
- `crates/app-infra/src/captured_frame_equivalence.rs` is the explicit seam for **Captured Frame Equivalence** policy: callers resolve the nearest earlier equivalent **Captured Frame** through `CapturedFrameEquivalenceResolver` instead of rebuilding quarantine/version/proof-match logic.
- The equivalence seam has two caller semantics: OCR/job fallback uses the nearest earlier equivalent **Captured Frame**, while the dashboard `duplicateOf` UI uses the earliest earlier equivalent frame in the duplicate chain so adjacent duplicates still point back to the first origin frame.
- **Captured Frame Equivalence Scope** is explicit at that seam: lookup is session-wide by default, but narrows to the same hidden segment workspace when the candidate frame path originates from a hidden segment workspace.
- Failed processing jobs are bounded-retried with a wall-clock backoff via the `processing_jobs.next_attempt_at` column. Retry now covers OCR **and** the audio processors (`audio_transcription`, `speaker_analysis`, `system_audio_speech_activity`), not OCR-only: `requeue_failed_job_within_attempt_cap` gates on a per-processor `FailureRetryPolicy` (`failure_retry_policy_for_processor`) — OCR backoff `OCR_RETRY_BACKOFF_SECONDS` `[30,120]s` capped by `OCR_FAILED_JOB_MAX_ATTEMPTS`, audio backoff `AUDIO_RETRY_BACKOFF_SECONDS` `[60,300]s` capped by `AUDIO_FAILED_JOB_MAX_ATTEMPTS` (audio jobs are minutes-long, so retries are spaced wider). The cap is gated on the `processing_jobs.failure_count` column, which counts only genuine failures (incremented by `mark_job_failed`), distinct from the total `attempt_count`. The automatic queue claim (`claim_next_queued_job_*`, ordered `id ASC`) skips jobs whose `next_attempt_at` is in the future so a deterministically-failing job does not re-claim ahead of fresh work every cycle; explicit `claim_queued_job(id)` (reprocess / direct `process_job`) bypasses the backoff. The generic `requeue_processing_job_in_transaction` resets `next_attempt_at` to NULL (immediately eligible), so backoff is failure-retry-only. Prefer wall-clock backoff over reordering the claim by `queued_at`: SQLite `CURRENT_TIMESTAMP` is second-granularity, so a sub-second reorder of a just-requeued job is unreliable.
- **Processing Job Reclamation** (`reconcile_orphaned_running_jobs` in `crates/app-infra/src/processing/store.rs`) requeues an **Orphaned Processing Job** (a row left `running` because its worker was aborted at app quit/crash) back to `queued` so it re-runs and still produces its result, instead of the old fail-all-running sweep. Abandonment does **not** spend a failure attempt (`failure_count` untouched); only a job reclaimed `RECLAIM_ATTEMPT_CEILING` (10) times without completing is left `failed` as a crash-loop backstop. It is safe only because it runs when nothing is executing: at startup (now inside the deferred-startup background thread via `AppInfra::run_startup_maintenance`, still before workers spawn — see the **Deferred Startup** bullet) and at graceful shutdown (`shutdown_background_workers_for_app_exit` in `apps/desktop/src-tauri/src/app_infra.rs`, after workers are aborted/awaited via `AppInfra::reconcile_orphaned_processing_jobs`). There is no live-session watchdog. See [ADR 0020](docs/adr/0020-reclaim-orphaned-processing-jobs-by-requeue.md).
- **Deferred Startup**: app startup is split so the window opens fast and heavy maintenance runs off the window-open critical path. `apps/desktop/src-tauri/src/app_infra.rs::initialize` does only `AppInfra::initialize_fast_with_processing_registry` (open the encrypted SQLite pool + build stores) and `manage`s state, so Tauri commands can serve queries immediately. Everything else — `AppInfra::run_startup_maintenance` (index backfills + orphaned-job reconciliation), hidden-segment workspace repair, frame-index sidecar conversion, OCR-disabled reconciliation, audio/speaker backfill, and spawning the processing/retention/repair workers — runs in `run_deferred_startup_blocking`, invoked from a dedicated `mnema-deferred-startup` background thread spawned in `lib.rs` setup **after** `open_startup_window`. Order is load-bearing and must be preserved: maintenance + hidden-segment repair complete before any worker or capture auto-start runs (repair assumes no capture is active, and reconciliation must precede workers per ADR 0020). Do not move maintenance/repair/worker-spawn back into the synchronous `initialize`/`setup` path, and do not start capture before the deferred maintenance completes. The startup delay was dominated by these scans (the SQLCipher DB open itself is ~200ms; `search.backfill_missing_projections` alone was ~700-950ms and used a `NOT EXISTS` that decrypted `result_text` for every row — now an indexed `LEFT JOIN` anti-join). Use `AppInfra::initialize` (fast + maintenance combined) for tests and the CLI sidecar; only the desktop app takes the split path.
- Native capture is macOS-oriented; many capture code paths and tests are behind `cfg(target_os = "macos")`.
- ScreenCaptureKit screen/system-audio liveness has two invalidation paths: AppKit `NSWorkspaceWillSleepNotification` clears the live screen side proactively, and the `crates/capture-screen` `SCStreamDelegate` also marks the stream dead on `stream:didStopWithError:` / `streamDidBecomeInactive:`; the `Recording Lifecycle` reconciles either signal by dropping only live screen/system-audio state while preserving microphone continuation.
- On system sleep, the `Recording Lifecycle` must preserve stale screen/system-audio paths inside `current_segment_output_files` even after clearing live `recording_file` / `system_audio_recording_file`, because wake recovery derives the interrupted segment outputs from that bookkeeping before starting the next segment.
- `apps/desktop/src-tauri/src/native_capture_output.rs` screen finalization now rejects existing-but-unopenable screen `.mov` files instead of treating mere file existence as success; invalid rotated segments should fail finalization rather than being committed and later breaking frame preview extraction.
- The dashboard inactivity debug surface has two different audio notions: `idleDebug.activitySources` carries threshold-qualified microphone/system-audio idle used for inactivity decisions, while `microphoneActivityLastUnixMs` / `systemAudioActivityLastUnixMs` are raw sample timestamps and should be labeled as samples rather than activity.
- Audio inactivity decisions run on the segment loop's coarse poll interval (currently 1s), so microphone/system-audio producers must preserve a peak-since-last-poll signal for inactivity evaluation; a single latest raw sample can miss brief real audio bursts.
- Per-family inactivity pause/resume decisions must treat unrequested sources as ineligible. Do not let microphone/system-audio family policies fall back to `internal_fallback` for unrequested sources, and keep screen guarded in both the family predicate and `pause_screen_for_inactivity_with_app_handle`; otherwise the `Recording Lifecycle` can emit repeated noop pause logs or latch incorrect paused state for sources that were never requested.
- Audio speech detection settings are shared under `audioSpeechDetection.detector` (`silero`, `webrtc`, `off`). Legacy `microphoneVadAdapter` may be read for compatibility, but saves/runtime normalization should use the shared setting; detector selection is exact and should not silently fall back from Silero to WebRTC.
- Inactivity activity mode is fixed to `system_input_or_screen_or_audio`. Legacy `activityMode` / `inactivityActivityMode` values can be deserialized, but settings validation should normalize saved/runtime settings to the fixed policy.
- System-audio transcription is gated by the `system_audio_speech_activity` processing job. System-audio segment commit should enqueue that job, not direct transcription; speech-activity completion may enqueue/requeue missing or failed `audio_transcription` jobs and keep speaker analysis chained through the transcription payload.
- Status-bar source toggles update persisted next-recording `RecordingSettings` only while stopped. The tray and frontend stay synchronized through `native_capture_session_changed` for start/stop session state and `recording_settings_changed` for settings changes.
- Live privacy refresh is coalesced through `apps/desktop/src-tauri/src/native_capture/privacy.rs`: settings/workspace events request a generation, wake `SegmentLoopControl`, and the segment loop owns applying completed filters. Static excluded-app-only refreshes use a metadata-free fast path and suppress the 1s fallback poll once the current request is satisfied until another settings/workspace event clears suppression.
- A live privacy-filter apply that fails because no capture display exists (display sleep, screen lock, lid close, monitor disconnect) is a transient liveness condition, **not** a privacy failure. `apply_privacy_filter_update` reports it with the distinct code `privacy_filter_display_unavailable` (`privacy::PRIVACY_FILTER_DISPLAY_UNAVAILABLE_CODE`), and the segment loop suspends screen/system-audio with `CaptureSuspensionKind::DisplayUnavailable` rather than the `PrivacyFilter` kind. The two suspension kinds share `PrivacyCaptureSuspension`/`privacy_capture_suspension` and the same recovery path (`attempt_privacy_suspension_recovery`), but differ in retry policy: `PrivacyFilter` is capped (`MAX_PRIVACY_CAPTURE_RECOVERY_ATTEMPTS`) and then escalates to a manual-restart notification, while `DisplayUnavailable` never escalates and keeps the session alive **even with no microphone** so screen capture auto-resumes when a display returns. A `DisplayUnavailable` suspension skips the in-flight-segment finalize (the `.mov` is already dead) and gates restart attempts behind `capture_screen::screen_display_available()` (a cheap `CGGetActiveDisplayList` check) plus a `DISPLAY_UNAVAILABLE_RECOVERY_INTERVAL` throttle, so it waits quietly without re-spamming ScreenCaptureKit `-3815` while the display is gone. See [ADR 0021](docs/adr/0021-recover-from-display-unavailable-as-transient-liveness.md).
- When the segment loop ends a session on its own (any internal `mark_runtime_session_failed`, not a user/command stop), the loop broadcasts the change after it exits — `emit_native_capture_session_changed` + `status_bar::refresh` — because user/command stops own that broadcast but an internal failure has no other announcer; without it the dashboard and native tray keep showing a running session that no longer exists.
- Sensitive Capture Protection V1 stays inside the app-only privacy model from ADR 0006 and ADR 0008. Do not add URL/domain/title/private-window/password-field/browser add-on filtering to live privacy; use App Privacy Exclusion, Browser Capture Disclosure, Exclude Current App, Pause Recording, and confirmed Delete Recent Capture recovery instead.
- ADR 0013 rejects browser-integrated credential-entry suspension and browser add-on metadata/safety plumbing in this branch. Browser Metadata Collection is native browser URL probing only unless a future ADR adds another adapter.
- Recommended App Exclusions are Rust-owned exact bundle-id catalog matches with finite curated categories. Keep Known Browser App values separate from the Sensitive App Recommendation Catalog, and do not default-recommend broad workflow apps such as System Settings, Terminal/developer tools, messaging, or email.
- Delete Recent Capture is not Retention Cleanup. It is a confirmed recovery action over recent wall-clock windows that deletes whole overlapping Capture Segments/Audio Segments and derived app data from Mnema's library without promising secure erase.
- User Capture Pause is distinct from inactivity pause: it keeps the Capture Session alive, finalizes the active segment, records no requested sources while paused, and resumes only from explicit user action.
- Ask AI delegates model/auth/session execution to the user's installed PI runtime, not Mnema-owned provider credentials. Current PI RPC does not expose host-executed tool callbacks, so Ask AI broker access should use a thin PI SDK/extension `defineTool` shim for the data tools `search`, `timeline`, and `show-text` (plus the presentation-only `reference_captures` tool — see ADR 0024's amendment), calling Rust Tauri commands in `apps/desktop/src-tauri/src/ask_ai.rs`; those commands must keep enforcement in `BrokeredCaptureAccess::execute_for_ask_ai` (the Ask-AI-specific seam that injects a synthetic All-Retained grant and rejects `open_in_mnema`), not the CLI's `execute_for_identity`, with PI's inferred identity. `open`/`open_in_mnema` remains an app-mediated dashboard handoff, not an Ask AI data tool. See ADR 0024.
- The Ask AI **seeded single answer** slice drives PI through a Node **SDK shim** (`apps/desktop/src-tauri/resources/pi-ask-ai-shim.mjs`), NOT `pi --mode rpc`. RPC mode cannot disable the coding-agent's builtin bash/file tools, so it cannot satisfy "those tools are never exposed"; the shim uses `@earendil-works/pi-coding-agent` with `noTools: "builtin"` (the coding-agent's builtin bash/file tools stay disabled and are never exposed) and an in-memory (ephemeral, no-disk) session. On top of that the shim DOES register three custom brokered tools — `search`, `timeline`, `show_text` — via `customTools` (SDK `defineTool`, TypeBox params from the `typebox` package); each calls back to the Rust host rather than touching the machine directly. The shim is shipped via `tauri.conf.json` `bundle.resources` and resolved at runtime as `resource_dir()/resources/pi-ask-ai-shim.mjs` with a `CARGO_MANIFEST_DIR/resources/...` dev fallback. New files added to `src-tauri/resources/` for bundling must be listed in `bundle.resources`.
- Ask AI shim invocation contract (Rust ↔ shim, owned by `apps/desktop/src-tauri/src/ask_ai/pi_agent_session.rs`): Rust spawns `node <shim>` (node resolved via `crate::app_infra::executable_in_shell_path("node")` — never bare `Command::new`, packaged macOS apps lack the Homebrew PATH), passes env `MNEMA_PI_EXECUTABLE` (resolved `pi` binary path, used by the shim to locate the SDK package) and passes through `PI_CODING_AGENT_DIR` only if already set. The protocol is bidirectional and stdin STAYS OPEN for the life of the session: Rust writes one `{"prompt":"…"}\n` line first, then writes `{"type":"tool_result","id":…,"ok":…,…}\n` lines back to the shim as tool calls resolve. The shim emits newline-delimited JSON on stdout — `{"type":"ready"}`, `{"type":"delta","text":"…"}`, `{"type":"tool_call","id":…,"tool":…,"params":…}` (a brokered tool the host must run), `{"type":"done"}`, `{"type":"error","message":"…"}` — and all diagnostics to stderr; `parse_shim_line` ignores blank/malformed/unknown lines and a clean/non-zero exit with no terminal line surfaces the stderr tail as the error. Tool calls are capped per question by `run_pi_ask_ai_session`'s `max_tool_calls`, resolved at start from the `access.askAiMaxToolCalls` setting (`crates/capture-types` `AccessSettings`, default 12, `0` = no cap → `usize::MAX` via `resolve_tool_call_cap` in `ask_ai.rs`); the Settings → Access "Limit tool calls per question" toggle + number input persists it. The Quick Recall model is likewise persisted as `access.askAiModel` (`crates/capture-types` `AccessSettings`, a `provider:id` string; `None`/empty → PI's default model) and read via `read_ask_ai_model` in `ask_ai.rs`, then passed to the shim as the `MNEMA_PI_ASK_AI_MODEL` env var; the Settings → Access model combobox is populated by the `ask_ai_list_models` command, which runs the shim in **list mode** (`MNEMA_PI_LIST_MODELS=1` → one `{"type":"models","models":[…]}` line, then exit) to enumerate the user's PI model registry, and free-form `provider:id` entry is allowed. Every tool call still rides `execute_pi_broker_request` → All-Retained broker scope + redaction/audit Rust-side (`open`/`open_in_mnema` is NOT an Ask AI tool and is rejected before the broker). Cancellation is cooperative (an `Arc<AtomicBool>` checked between output lines, then `child.start_kill()`), so a cancel during model idle is only observed on the next token.
- Ask AI streaming Tauri surface (frontend ↔ Rust): commands `get_pi_runtime_status`, `ask_ai_availability` (`{ available, reason? }`; returns Ok with `available:false` for the normal off/unready case rather than erroring — reasons `ask_ai_disabled` or the `PiRuntimeStatus.reason`), `ask_ai_list_models` (enumerate selectable PI models — see the model-selection note in the shim bullet above), `ask_ai_start({ conversationId, question, seedQuery, priorTranscript? })`, `ask_ai_followup({ conversationId, question })` (routes a raw follow-up into the resident PI session of an existing thread — NO seeding and NO `seedQuery`; the answer streams back over the same events; an absent live session errors `Ask AI conversation is no longer active`, which the frontend treats as a dead thread and resurrects via `ask_ai_start`+`priorTranscript`), and `ask_ai_cancel({ conversationId })`, all registered in `lib.rs`. Streaming is event-based, NOT a command return value: snake_case events `ask_ai_status` (`phase: "seeding" | "thinking" | "tool"`, plus `seededResultCount`; the `tool` phase additionally carries `tool` + `params` to drive the Quick Recall tool-activity working-line UI), `ask_ai_delta` (`{ conversationId, text }`), `ask_ai_done`, `ask_ai_error` (`{ conversationId, message }`); every payload carries `conversationId` and the frontend ignores stale ones. Seeding runs the broker search best-effort (broker error/empty → proceed unseeded, never abort the ask). The conversation is disk-ephemeral — nothing persists, and a full **Quick Recall** close / app exit tears the thread down — but an **unseen** thread (still streaming, or finished but never viewed while focused) is NOT cancelled on dismiss/blur: `dismiss_quick_recall_window` defers the teardown decision to the panel, the answer finishes in the background, and a re-summon resumes it. A finished-but-unseen thread keeps its resident PI session for a fixed 30-minute frontend-owned cap (`ASK_UNSEEN_EXPIRY_MS` in `quick-recall/+page.svelte`), then cancels the helper while leaving the transcript readable (thread marked `expired`). A follow-up on a **dead** thread (expired, or last turn errored) RESURRECTS rather than follows up: a fresh `ask_ai_start` carrying `priorTranscript` (completed Q/A only, no tool logs/Answer Sources), which `ask_ai.rs` formats into an `ASK_AI_PRIOR_TRANSCRIPT_CHAR_CAP` (12k-char) oldest-first-trimmed context block prepended to the first prompt; `seedQuery` stays the bare question so seeding is never polluted by the transcript. Absent/empty `priorTranscript` behaves exactly as a plain start. See [ADR 0027](docs/adr/0027-ask-ai-threads-complete-in-background-and-resurrect-from-transcript.md).
- `install_from_tar_bz2` (`apps/desktop/src-tauri/src/speaker_analysis_models.rs`) extracts tarball entries by matching whole final path components via `Path::ends_with("model.onnx")`. This means a quantized sibling like `model.int8.onnx` is correctly skipped (different component) and only `model.onnx` is extracted — relevant when adding a `tar.bz2`-packaged speaker segmentation preset whose archive ships multiple `*.onnx` files. Set each descriptor's `required_files` + `sherpa_params.segmentation_relative_path` to the destination subdir the extractor writes to.

## Verification
- UI-only changes: `bun run check`.
- Rust changes in one crate: start with `cargo check -p <crate>` or `cargo test -p <crate>`.
- Tauri wiring or cross-crate Rust changes: run `cargo check --manifest-path apps/desktop/src-tauri/Cargo.toml`.
- Cross-stack or settings/storage changes: run both `bun run check` and `cargo check --manifest-path apps/desktop/src-tauri/Cargo.toml`.
- Focused tests in `apps/desktop/src-tauri/src/lib.rs` may need `cargo test --manifest-path apps/desktop/src-tauri/Cargo.toml --lib <test-filter>`; without `--lib`, filtered Tauri tests may not run as expected.
- Status-bar menu model changes: run `cargo test --manifest-path apps/desktop/src-tauri/Cargo.toml --lib status_bar`.
- Verifying the `audio-transcription` crate's `local-whisper` feature requires `cmake` in `PATH`, because `whisper-rs-sys` builds bundled `whisper.cpp`/GGML artifacts.
- Any `cargo check`/`cargo test` against `apps/desktop/src-tauri/Cargo.toml` needs the `mnema-cli` sidecar present, or the Tauri build script fails with `resource path 'binaries/mnema-cli-<triple>' doesn't exist`. `bun run tauri -- dev`/`build` build it automatically, but a bare `cargo` invocation does not — run `bash scripts/prepare-mnema-cli-sidecar.sh debug` (or `release`) first to compile/copy it into `apps/desktop/src-tauri/binaries/`.

## Workflow
- For desktop confirmations, alerts, and file dialogs, use Tauri dialog APIs/plugins (for example `@tauri-apps/plugin-dialog` in Svelte) instead of browser-native `window.confirm`, `window.alert`, or ad hoc browser dialogs.
- When new repo-specific behavior, commands, structure, or gotchas are discovered during a change, ask the user whether that context should also be added to `AGENTS.md`.
