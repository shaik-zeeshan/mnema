# Ask AI multi-turn reuses one live PI session for the whole thread

**Quick Recall**'s **Ask AI** is no longer a one-shot question/answer; it is an ephemeral multi-turn **Ask AI Thread**. The whole thread is backed by **one** long-lived Node shim child and **one** PI Agent SDK session that stays resident for the life of the thread. Follow-up questions are fed into that same session as additional `{"prompt":…}` lines; the session retains prior turns in-memory, so each follow-up is answered with the earlier turns already in context. Mnema does **not** replay the transcript or re-seed context to the cloud per turn — context accumulates inside PI, and only the new turn's text is sent on each follow-up.

This **reverses** the deliberate "first prompt only" constraint the shim shipped with under [ADR 0024](0024-ask-ai-uses-pi-tool-shim-over-installed-runtime.md). That slice answered exactly one question per child: the shim read only the first `{"prompt":…}` line ("Only the first prompt matters; ignore any later prompt lines") and the SDK subscribe handler ended the **process** on the first `agent_end`. The same long-lived child the shim already runs is now kept alive across turns instead: each `agent_end` ends the **current turn** (emit a turn-level done and re-enable the composer), the session stays subscribed, and the next `{"prompt":…}` line starts the next turn. The session ends only when stdin closes or the child is killed. This builds on the **Quick Recall** Ask AI pivot established in [ADR 0025](0025-quick-recall-search-syntax-uses-launcher-native-autocomplete.md) (`Ctrl`/`Cmd+Enter` from **Quick Search** into the ask), which seeds the first turn.

**Considered Options**

- **Reuse one live PI session per thread (chosen).** Keep the shim child + PI session resident; feed follow-ups as new prompt lines on the same session, letting PI hold the conversation history. The agent reaches new context through its own `search`/`timeline`/`show_text` broker tools, not through replayed prior answers. This is the cheapest path to coherent follow-ups (no transcript bookkeeping on our side, no re-uploading prior turns) and keeps the redaction/retention boundary exactly where [ADR 0022](0022-ask-ai-sends-redacted-capture-context-to-cloud-agent.md) and [ADR 0024](0024-ask-ai-uses-pi-tool-shim-over-installed-runtime.md) put it.
- **Re-seed each turn into a fresh session, replaying the transcript.** Rejected. It would re-send all prior turns to the cloud on every follow-up (more outbound capture-derived content per turn, against the minimize-what-leaves principle of ADR 0022), require Mnema to own and stitch a transcript, and lose the agent's accumulated tool state. The only thing it buys — a stateless child between turns — is not worth re-paying the seeding cost each turn.
- **Spawn a new child per follow-up.** Rejected for the same reasons plus per-turn Node/PI startup latency; it also discards the in-process agent loop's memory of earlier `search`/`timeline` results.

**Consequences**

- **`agent_end` ends a turn, not the process.** The shim's per-turn handler emits a turn-level done and keeps the session subscribed; the child lives until stdin closes or it is killed. The original ADR 0024 "first prompt only" behavior is explicitly retired.
- **`conversationId` now identifies the whole thread/session**, not a single Q&A. The stream events (`ask_ai_status`/`delta`/`done`/`error`/`source`) keep carrying it; the frontend appends deltas to the current (last) turn keyed by that id and ignores stale ids. `ask_ai_done` now means **this turn finished** (re-enable the follow-up composer), not the thread is over.
- **The tool-call cap is per turn, not a per-thread budget.** `resolve_tool_call_cap` (from `access.askAiMaxToolCalls`, default 12, `0` = uncapped) is applied per `session.prompt()` turn and resets each turn, so a thread of N follow-ups gets up to N×cap data-tool calls. `reference_captures` stays **exempt** from the cap, per the [ADR 0024](0024-ask-ai-uses-pi-tool-shim-over-installed-runtime.md) amendment — it reaches no data, so counting it would let a long session starve the model's ability to declare its **Answer Sources**.
- **First-turn-only seeding.** The broker seed search and the natural-language scope suffix derived from active **Quick Recall Filter Chips** ([ADR 0025](0025-quick-recall-search-syntax-uses-launcher-native-autocomplete.md)) apply **only to the first turn**. Follow-ups carry the raw typed question with no `seedQuery` and skip the broker seed search; the agent uses its own tools for anything new. There is no re-scope or attach surface mid-thread in V1.
- **Lifecycle joins the existing window-reuse model.** The thread lives only while **Quick Recall** is open, surviving a hide/re-summon inside the existing `IDLE_CLEAR_MS` (5s) idle window the same way **Quick Search** state already does. Hiding starts the idle timer; re-summon within the window **resumes** the live thread (the PI child stays resident, an in-flight stream keeps buffering, the frontend reattaches). Idle-window expiry while hidden, Escape-to-search, or cancel **tears down** the child and discards the thread.
- **Resident-while-hidden is accepted.** A direct consequence of the above is that the PI child + cloud PI session stay resident for up to the idle window **even while the Quick Recall panel is invisible**. This is named and accepted: it is consistent with how **Quick Search** state already persists across a quick hide, and the window is bounded (the child is killed at idle-window expiry). It is not a leak — it is the same bounded-resume contract applied to a longer-lived child.
- **Every teardown path must route through cancel.** Because the child now outlives a single answer, all teardown paths — Escape-to-search, idle-window expiry while hidden, blur/quick-hide close, and app exit — must route through the cancel path (`cancelActiveAsk`/`ask_ai_cancel`) so no orphaned Node child outlives the panel. Cancellation stays cooperative: an `Arc<AtomicBool>` checked between output lines, then `child.start_kill()`. There is no per-turn "stop generating" in V1 — while a turn streams the composer is disabled and the only interrupt is Escape (hard-cancel the whole thread); a per-prompt abort that keeps the thread alive is deferred (it needs the PI SDK per-prompt abort signal rather than killing the child).
- **Nothing persists.** The thread is in-memory only: no conversation store, no thread/recent-threads list, no resume across a full **Quick Recall** close. This keeps Quick Recall's documented ephemerality and privacy posture.

**Out of scope**

- Persisted chat history, a thread/conversation list, or resuming a thread across a full **Quick Recall** close.
- A per-turn "stop generating" that keeps the thread alive (deferred; needs the PI SDK per-prompt abort signal).
- Attaching a specific **Quick Search** result into the thread, or re-scoping with **Filter Chips** mid-thread.
- Re-seeding the broker search on follow-up turns.
- Any change to redaction, retention, broker scope, or the `reference_captures` presentation contract from [ADR 0024](0024-ask-ai-uses-pi-tool-shim-over-installed-runtime.md).
