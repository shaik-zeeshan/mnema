# System audio is an independent capture family on Core Audio process taps

## Status

Accepted 2026-07-15 (design grill). Ships with the CATap swap PR; until then the code still runs the SCK path this ADR replaces.

## Context

System audio has always ridden the screen capture stream: one shared ScreenCaptureKit `SCStream` delivers both video and system audio (`crates/capture-screen/src/lib.rs`), so system audio inherits every screen coupling — it suspends when the display sleeps (ADR 0021), its resume logic asks whether the screen is paused, its Settings toggle is disabled without screen capture, and any restart to fix audio restarts screen recording too.

On 2026-07-15 the SCK audio tap died in the field mid-session: callbacks stopped entirely for 34+ minutes while audio was audibly playing, with zero errors in the unified log. This SCK failure mode (delivery just stops) is industry-wide (OBS, screenpipe, pyobjc reports), and the only SCK-level remedy — restarting the stream — would restart *screen* capture to fix *audio*.

Core Audio process taps (`CATapDescription` + aggregate device, macOS 14.2+) capture system audio with no display or screen-stream dependency. cidre (already the foundation of every capture crate) covers the full API behind its `core_audio` feature — zero new dependencies. CATap has its own known failure mode on macOS 26 (Apple forums 825780): the IOProc keeps firing but delivers all-zero samples for stretches, triggered by sample-rate renegotiation, Bluetooth sleep/wake, or long uptime; the only reliable recovery is a full teardown-and-rebuild of tap + aggregate. For a passive recorder this is a manageable trade: during all-zero delivery there is nothing worth recording, so a rebuild costs no real audio — unlike an SCK restart, which tears down a healthy screen stream.

One coupling is load-bearing and easy to miss: the SCK content filter (`with_display_excluding_apps_excepting_windows`) excludes privacy-listed apps' **audio** as well as their windows. Any replacement must preserve that, or privacy-excluded apps silently start being recorded.

## Decision

System audio becomes its own capture family, structurally a sibling of the microphone: own crate, own session, own inactivity lifecycle, no screen dependency.

- **Clean swap, no SCK fallback.** The SCK audio path (audio output on the shared stream, `set_captures_audio`, the audio callback arm) is deleted. Rollback insurance is a revert-friendly PR plus fast point releases, not a second maintained audio path — a fallback would force keeping all the screen-coupled machinery whose deletion is half the project's value, and SCK is not a "safe" fallback anyway (its dead-tap mode is field-confirmed).
- **New crate `crates/capture-system-audio`**, mirroring `capture-microphone`'s shape and feeding the shared `capture-writers` (m4a/AAC at the existing planner paths, so transcription/VAD/speaker/retention/search are untouched). It alone enables cidre's `core_audio` feature. `capture-screen` only loses code.
- **Conservative aggregate shape** (AudioCap/AFFiNE): global tap excluding a process list, aggregate device with the real default-output device as main sub-device, drift compensation on, unique private UID per instance, raw IOProc on a non-nil dispatch queue. The leaner tap-only aggregate (cpal's shape) has documented zero-sample failures; every production implementation rebuilds on device change regardless, so the leaner shape buys nothing.
- **The tap exclude list = Mnema's own process + privacy-listed apps.** Excluding self is new deliberate behavior (a recorder capturing its own notification sounds is noise). Excluding privacy-listed apps is **parity, not a feature** — SCK's content filter already silenced them. The list is process-object-based, so it is dynamic: a privacy-list edit mid-recording, or an excluded app (re)launching an audio process, triggers a tap rebuild. A listener on the system audio-process list, filtered to excluded bundle IDs, bounds the churn.
- **Writer format follows the tap** (mic's proven pattern: derive format from observed sample buffers, lazily create the writer once stable). No resampler — nothing downstream reads the recorded rate; transcription/VAD/diarization decode and resample themselves. The tap's ASBD is device-dependent (44.1 ↔ 48 kHz) and changes across rebuilds.
- **Rebuild is the one recovery mechanism**, triggered by: default-output-device change or device death (property listeners), privacy/process-list changes, and a **zero-watchdog** — sustained all-zero delivery for ~30 s triggers a rebuild, with exponential backoff (30 s → 60 s → … capped ~10 min) that resets on any non-zero sample, because real silence also produces zeros. The watchdog runs **even while system audio is paused for inactivity**: the resume trigger is "sound detected," and a wedged tap can never deliver the sound that would wake it (exactly how the SCK bug trapped a live session). Every rebuild finalizes the current segment and starts a fresh one via existing rotation machinery — no splicing tap generations into a live writer, which sidesteps mid-writer format changes and timestamp discontinuities for the price of a routine segment boundary.
- **Permissions use the official path only.** CATap has its own TCC category ("Screen & System Audio Recording"), separate from the Screen Recording grant; the prompt fires on first tap read; there is **no API to query authorization**. No private-SPI pre-check. `NSAudioCaptureUsageDescription` is added; existing users get the OS prompt at their next recording start after updating; onboarding's system-audio step gains a real Grant button that briefly runs a throwaway tap to trigger the prompt. Denial is inferred, never known: if system audio is enabled and every session since the swap delivered only zeros, a dismissible "system audio may be blocked" hint deep-links to the privacy pane. Permission display becomes *not yet requested / assumed working / possibly blocked*, replacing the current alias onto the screen permission.
- **Display independence.** System audio records through display sleep/lock/disconnect, like the microphone (ADR 0021 amended: its suspension becomes screen-only). Low-disk suspension (ADR 0040) still covers all sources — that one is about the disk. All screen-liveness gates on system audio (resume action, re-anchoring, watchdog gating, the Settings toggle's `disabled` coupling, onboarding's "requires Screen capture") are severed. **System-audio-only recording sessions are allowed** — forbidding them would mean writing new coupling into a decoupling project.
- **Out of scope:** live VAD for system audio (mic-style tail trim) — RMS inactivity already works continuously and the post-hoc speech pipeline is untouched; lowering the macOS 15.0 runtime gate (CATap would allow 14.2+, deferred).

## Alternatives Rejected

- **Keep SCK and restart the stream on watchdog trip.** Restarts screen recording to fix audio; punishes the healthy source and is visible to the user. Also inherits every future SCK regression on a stream we need for video anyway.
- **SCK fallback behind a setting for one release.** Protects against the wrong risk (the zero-bug's mitigation is the rebuild watchdog, not SCK) and forces both audio paths — including the screen-coupled inactivity machinery — to stay alive in two modes.
- **`objc2-core-audio` bindings.** cidre is already the foundation of every capture crate and its `core_audio` feature covers the entire surface; a second binding stack for one crate is pure overhead.
- **Resample to pinned 48 kHz/stereo.** New DSP code and a new artifact source, preserving a uniformity nothing consumes.
- **Continue the writer across rebuilds (new tap generation into the same file).** Invites PTS discontinuities and cannot survive a format change; segment boundaries are already routine (5-minute cap).
- **Private TCC SPI to pre-check/pre-prompt permission** (AudioCap's approach). Viable for direct distribution but fragile across macOS updates on a privacy-adjacent surface; the official prompt-on-first-read plus a denial heuristic is honest and unbreakable.

## Consequences

- An audio failure is now recoverable in-place, invisibly, without touching screen or mic — and detection no longer depends on the failing source proving its own death (the zero-watchdog assumes guilt after sustained zeros, and a rebuild during zeros is lossless by construction).
- Every existing user sees one new macOS permission prompt after updating; a denied user records zero system audio with no OS-level signal, mitigated only by the heuristic hint.
- CATap's macOS-26 all-zero wedge is not eliminated — it is converted from "silent data loss until manual restart" into "bounded gap, auto-rebuilt." Rebuild storms are bounded by backoff; the soak before release watches for them.
- ADR 0021's display-unavailable suspension narrows to screen only; `apps/desktop/src-tauri/CONTEXT.md`'s "screen and system audio share the screen capture backend" language, `SUPPORTS.md`'s "requires screen capture" rows (and its Windows independent-source TODO), and CLAUDE.md capture invariants all need updating when the swap lands.
- Verification: the watchdog/backoff state machine and denial heuristic are pure-logic unit tests; device-switch, Bluetooth-drop, `killall coreaudiod`, and `tccutil reset SystemAudioCaptureRequests` are scripted manual drills; **privacy-exclusion parity (audio from an excluded app, including its helper processes, stays silent — baselined against SCK before the swap) is a merge blocker**; a multi-day soak on macOS 26 with rebuild events logged gates the release.
