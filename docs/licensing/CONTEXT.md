# Licensing Context

How Mnema is monetized: a one-time **License** with a paid **Update Window**, a time-limited **Trial**, and the offline, account-less machinery that verifies both. This context is cross-cutting — license/trial state and its keychain store live in `crates/app-infra`, the Tauri commands and gate live in `apps/desktop/src-tauri`, and the buy/trial UI lives in `apps/desktop/src`.

Everything here is **local-first and account-less** *for the app and the user*: the app **activates once per machine, then never phones home**. The payment platform (Polar, as Merchant of Record) only takes the money; **Fulfillment** is **licensegate** (the seller's standalone licensing platform at `license.mnema.day`, ADR 0054), which mints the signed key, emails it (Polar cannot carry our dynamically-minted key), and answers one **Activation** request per machine with a signed **Activation Receipt**. After that single exchange, the app verifies key + receipt entirely on-device against a baked-in public key — no heartbeat, no expiry, no runtime calls.

**Privacy commitment (public):** the activation request contains exactly two things — the license id and a salted, irreversible hash of a hardware identifier. No name, no email, no device name, no OS version, no telemetry. The server stores, per license: the set of those hashes (max 3), a lifetime count, and the last reset time; no IP logs retained. After activation succeeds, the app never contacts the activation endpoint again for that machine. (The CRL fetch remains an anonymous GET of a public file.)

## Language

**Capture** (existing capture-pipeline term):
Forward recording of screen/audio into Mnema. This is the *paid capability* — the thing a License unlocks and the thing that stops when unlicensed.

**License**:
A one-time purchase that grants permanent **Capture** rights on the builds it covers, plus updates released within its **Update Window**. On any given machine those rights are unlocked by an **Activation Receipt** (or, briefly, the **Provisional Window**). The signed payload carries the buyer's `name` and `email`, shown as "Licensed to ⟨name, falling back to email⟩" in Settings — ink on the certificate as a social deterrent, never verified.
_Avoid_: subscription, seat.

**Update Window**:
The dated period after purchase during which newly-released builds are covered by a **License** (launch length: 1 year). Encoded in the signed key as an `update_through` date. Renewing extends it.
_Avoid_: expiry, license expiry (the *app* never expires — only new-build coverage does).

**Perpetual Fallback**:
The rule that an owner permanently keeps the right to *run* the newest build released within their **Update Window**, even after it lapses — they simply stop receiving newer builds. "Perpetual" means perpetual **use**, not perpetual **updates**.

**Trial**:
A time-limited grant of full **Capture** rights for evaluating Mnema before any purchase (launch length: 30 days). No account, no card. *(Amended 2026-07-16, licensegate migration:)* the trial is now a **server-issued trial key**, requested at **first successful Capture** — one trial per machine, enforced server-side. The clock starts when the server issues the key. If the request fails (offline, server down), Capture still starts immediately and the app retries quietly; after 7 days of never reaching the server (mirrors the **Provisional Window**), Capture pauses until one issuance succeeds. Nobody's first capture ever blocks on a network error.

**Read-Only Mode**:
The state after **Trial** expiry (and before purchase) where all already-recorded history stays fully browsable, searchable, and Ask-AI-able, but new **Capture** is disabled. Buying a **License** re-enables **Capture**.
_Avoid_: hard-lock, paywall (Read-Only Mode is explicitly *not* a lock-out).

**Fulfillment**:
The seller-side step that turns a paid Polar webhook into a delivered key. *(Amended 2026-07-16, ADR 0054:)* Fulfillment is **licensegate** — the seller's standalone licensing platform (one deployment at `license.mnema.day`, two products: `mnema` for release builds, `mnema-dev` for debug). It verifies the Polar webhook, mints the Ed25519-signed key (private key held server-side, never on the build machine), emails it via Resend, and serves **Activation**, server **Trial** issuance, checkout **claim**, device **Reset**, validate, and the CRL. The desktop app consumes it through the pinned `licensegate` client crate; after the once-per-machine exchange it only verifies signatures offline.
_Avoid_: license server (there is no per-check runtime validation — **Activation** is once per machine, ever, not a recurring check), fulfillment worker (the Cloudflare Worker is deleted).

**Renewal**:
A separate one-time Polar SKU that extends an owner's **Update Window** by another year. *(Amended 2026-07-16, ADR 0054:)* the renewal product maps to licensegate's `action: renew` — it **extends the buyer's existing license** (same license id, `updates` entitlement pushed out a year) and re-mints + re-emails the key, rather than minting an unrelated fresh key. Renewal-for-existing-owner enforcement and delivery live in licensegate's provider pipeline; ownership truth stays in Polar (the Merchant of Record). Orphan renewals (no matching license) surface in the licensegate dashboard for manual resolution. *(Amended 2026-07-17:)* a renewal reaches an already-activated machine through **Receipt Refresh** — the extended dates arrive on the next idempotent re-activation, never via a new key.

**Receipt Refresh**:
The event-driven re-activation that delivers server-side license changes (a **Renewal**, an admin extension) to an already-activated machine, replacing its stored **Activation Receipt** with a fresh one carrying current dates. Refresh is **never scheduled while the license is healthy** — an in-window activated machine sends nothing, ever. It fires on: (1) key install (paste or claim — the existing silent re-activation), (2) return from the in-app **Renewal** checkout (`mnema://license/renewed` deep link → poll refresh ~2s until the extended window lands, give up politely after ~1 min; the renewing machine already holds the key, so renewals never ride the claim flow — claim is key *delivery*, and server-side claim only resolves the original mint's checkout ref anyway), (3) user demand — a visible **"Refresh license status"** button in Settings → licensing, available from **any** state including healthy (user-initiated contact never counts against the no-phone-home promise; nobody waits out the cadence), with the Read-Only screen's **"Re-check license"** button as the same action, plus opening the licensing section of Settings while unhealthy refreshes automatically — and (4) a background cadence **only while a paid license's Update Window is lapsed**: every 4 hours for the first 14 days after lapse, then daily, stopping the moment a refresh reports the window open again. Expired trials and Revoked licenses get **no** background cadence — those populations are unbounded (every abandoned trial, forever) and the server almost never changes them; the rare support gesture (comped trial days, an un-revoke) propagates via the Re-check button. Provisional keeps its pre-existing 7-day activation retry loop unchanged. A failed refresh changes nothing — staleness never locks; the stored receipt remains offline-forever valid.
_Avoid_: heartbeat, license check-in (refresh only runs from an unhealthy state and only ever *improves* it), daily receipt refresh (the licensegate integration doc's unconditional daily cadence — deliberately not adopted; see the privacy commitment).

**Activation**:
The single online exchange that binds a **License** to a machine: the app sends the license id, a generic hardware model label (*amended 2026-07-17, ADR 0055* — "MacBook Air (M2, 2022)", never the personal computer name), plus a salted, irreversible hash of a hardware-stable machine identifier (macOS Hardware UUID — survives factory reset; salt derived from the license id so the same machine always hashes the same) to licensegate, which — if the license is not **Revoked** and under its **Device Cap** of 3 machines — returns an **Activation Receipt**. Activation is **mandatory** (a key without a receipt ends in **Read-Only Mode** after the **Provisional Window**) and **idempotent** (re-activating a known machine returns a fresh receipt and consumes nothing — factory reset, reinstall, and re-paste are free). **Comp Keys are not exempt.** *(Amended 2026-07-16, ADR 0054:)* the server-issued **Trial** rides the same rails — issuance chains straight into activation, so the trial-issuance request is now the first byte Mnema's licensing sends anywhere (its machine hash is salted by product slug, unlinkable to the license-id-salted activation hash by construction).
_Avoid_: registration, sign-in, heartbeat (there is no account and no recurring check while healthy — see **Receipt Refresh** for the only re-contact, from unhealthy states).

**Activation Receipt**:
An Ed25519-signed, domain-separated document binding one **License** to one machine hash, minted by the **Activation** endpoint and verified offline against the same hardcoded public key **forever** — no TTL, no re-check, no expiry. A receipt never outranks the **Revocation List**: receipt says "this machine may use this license," CRL says "this license is dead" — CRL wins.

**Provisional Window**:
Up to **7 days** during which a stored, signature-valid key grants **Capture** while the app retries **Activation** quietly in the background. *(Amended 2026-07-16, ADR 0054: the window anchors on `first_seen_at` — stamped when the key is first stored, write-once per license id, rollback-guarded — superseding "first failed activation attempt"; the same moment in practice, since storing a key always fires an immediate attempt.)* Pasting while online consumes nothing — a paste that activates immediately never opens the window; once open, the window runs on the clock regardless of why later retries fail (server down and user offline are deliberately not distinguished — the residual self-heals on the next successful reconnect). Consumed **per license id**, recorded in the OS keychain with the same max-timestamp-ever-seen rollback guard as the **Trial** — re-pasting the same key grants no new window; a genuinely different purchased key gets its own. If the window runs out unactivated, the app drops to **Read-Only Mode** ("couldn't confirm your license — connect once to finish activation") until an activation succeeds.
_Avoid_: grace period (the window only opens when an activation actually fails, never from paste alone), offline mode.

**Device Cap / Reset**:
A **License** activates at most **3 machines** (a lifetime *set* of machine hashes, not a counter). At-cap refusal is never a dead end: *(amended 2026-07-16, ADR 0054)* the over-cap screen carries an in-app **"Free up my devices"** button — a key-authed `POST /v1/reset`, rate-limited to once per 30 days — that empties the slot set and auto-retries activation (a cooldown shows the server's retry date); the refusal also carries a `reset_url` support note and a buy-another-license link. Reset **cannot** kill already-issued receipts (they verify offline forever). Egregious leaks are caught instead by the lifetime distinct-machine count — the abuse telemetry that feeds the **Revocation List**. Settings shows the device **count** (`{used, cap}`) only — no per-device dashboard, no device names sent or stored.

**Revocation List (CRL)**:
A small signed document listing the license ids of fully-refunded orders **and seller-revoked leaked keys** (e.g. a license whose lifetime activation count betrays public sharing, or a leaked **Comp Key**), published per-product by licensegate (`GET /v1/crl/{slug}`) and fetched anonymously by the app (no license id or identifier is ever sent). A key on the list is **Revoked**: authentic but no longer valid. The CRL is the only channel that reaches **already-activated** machines; the **Activation** endpoint additionally refuses new activations of Revoked ids, so a dead key can never plant itself on a new machine.
_Avoid_: license server, blacklist, deactivation.

**Revoked**:
The state of a **License** whose order was fully refunded, whose key demonstrably leaked (activation telemetry), or a **Comp Key** the seller withdrew. A Revoked key is rejected exactly like **Trial** expiry — the app enters **Read-Only Mode**, live if the key is in active use. User-facing copy says "revoked", never "refunded". Partial refunds never revoke.

**Comp Key**:
A **License** the seller gifts outside Polar (press, friends, self) — *(amended 2026-07-16, ADR 0054)* admin-minted in the licensegate dashboard, no order behind it. It is a *real* License: **Capture** is permanent, only its **Update Window** is set short (e.g. 90 days). It is identifiable by its license record and hand-revocable via the **Revocation List** if it leaks. Strangers evaluating Mnema use the **Trial**, not a Comp Key.
_Avoid_: trial key, demo key (a Comp Key never expires back to Read-Only Mode — it is a gift, not an evaluation window).

## Relationships

- A **Trial** grants **Capture** for a fixed window, then transitions the app to **Read-Only Mode**.
- A **License** grants **Capture** permanently and supersedes both **Trial** and **Read-Only Mode**.
- **Read-Only Mode** disables **Capture** but never restricts reading already-recorded history.
- An **Update Window** lapse enforces **Perpetual Fallback** via the **auto-updater** (it declines builds dated after `update_through`), *not* via any runtime lock — a lapsed owner keeps full **Capture** on their covered build.
- A paid Polar webhook triggers **Fulfillment** (licensegate), which mints and emails the key; a **Renewal** SKU extends the same license's `updates` entitlement and re-mints the key.
- A **full** refund (Polar order status `refunded`) puts the order's license id on the **Revocation List**; a **Revoked** key drops the app into **Read-Only Mode** through the same seam as **Trial** expiry. Partial refunds (`partially_refunded`) never revoke — they are goodwill, not an unwound sale. *(Amends the original "keys are non-revocable" decision in ADR 0045.)*
- A **License** grants **Capture** on a machine only through an **Activation Receipt** or its **Provisional Window**; no receipt and no window → the app behaves as unlicensed.
- **Activation** state lives per license in licensegate: `{ machine hashes (≤3), lifetime count, last reset }`. A lost-key **re-mint** and a **Renewal** both re-key the *same* license (same license id, same activation set) — an already-activated machine re-pastes and consumes nothing, and a revocation always covers every re-minted copy. *(Amends the pre-0054 "renewals carry their own slots" derivation scheme.)*
- **Staleness never locks — scoped to *after* activation**: a missing, unreachable, or stale CRL means the license stands, and a machine holding an **Activation Receipt** is offline-forever safe. A machine that never completed **Activation** is a different case — its **Provisional Window** ends in **Read-Only Mode**, because the license isn't fully established yet. The accepted residual: a refunder who keeps an *activated* machine offline keeps a frozen build.
- ~~The **Trial** is 100% serverless: keychain state only, no activation, no server contact of any kind before a key is pasted.~~ *(Superseded 2026-07-16: trials are server-issued via licensegate; first capture triggers the issuance request, with a 7-day offline grace before Capture pauses.)*

## Documented edge case — fresh install after Update Window lapse

Enforcement normally lives entirely in the auto-updater: a non-renewing owner is simply never offered a newer build, so they never hit a wall. The exception is a **clean install on a new machine after the Update Window has lapsed** — the download site serves the *latest* build, which the owner's key does not cover.

Resolution (for the plan): the app compares its own build date against the key's `update_through` at launch. If the installed build is newer than the window (owner, not trial), it does **not** hard-lock — instead it directs the owner to the newest build their **License** covers (kept downloadable), or to renew. Old covered builds must therefore remain downloadable indefinitely. This is the single place Perpetual Fallback needs a runtime check rather than an updater gate; it must never degrade already-recorded history, matching **Read-Only Mode**'s never-lock-existing-data rule.

## Flagged ambiguities

- **"stop capture" is overloaded.** Capture can stop for two unrelated reasons: (1) **Capture Suspension** — a *transient liveness* condition (display unavailable, low disk) that self-heals when the condition clears (see ADR 0021, 0040); (2) **Read-Only Mode** — a *licensing* state that does not self-heal and is cleared only by purchasing a **License**. These are distinct concepts and must not share a code path or a user-facing message.
- **Two different lapse gates.** "Unlicensed" conflates (A) **Trial expiry** → Read-Only Mode, and (B) **Update Window** lapse for a paying owner → *nothing is taken away*; they keep their owned version fully functional and only stop receiving new builds. Only (A) triggers Read-Only Mode.
- **"Trial start" resolved.** Ambiguous between first launch and first capture — resolved: the **Trial** clock starts at **first successful Capture**. Setup/permission time before any recording does not consume trial days.

## licensegate migration decisions (2026-07-16)

Resolved in the migration grill; the successor/amending ADR to 0053 will carry the full rationale.

- **Trials move server-side** (see amended **Trial** term above): issued by licensegate at first Capture, clock starts at issuance, capture never blocks offline, 7-day unreachable-server ceiling.
- **Mnema owns the clock; licensegate owns the state machine.** The existing max-timestamp-ever-seen rollback guard stays in the app and feeds the *guarded* time into the crate's `evaluate()` — clock rollback still cannot stretch a Trial or an Update Window. The crate's `clock_tampered` flag is diagnostic only, never a lock (a broken clock never punishes a paying customer).
- **"Capture is forever" is encoded in server product config, not adapter code.** Paid keys mint two entitlements: `app` perpetual (no end date — licensegate native) and `updates` fixed 365 days; because `app` never lapses, the crate's `Expired` state is unreachable for a paid key by construction. Trial keys mint `app` with a 30-day end date, so **Expired → Read-Only Mode is exactly the trial-expiry path**. `updates.expires_at` replaces `update_through` for update-eligibility gating. One pinned Mnema test asserts a perpetual-`app` key can never evaluate Expired (guards against server-side config fat-fingering).
- **`first_seen_at` is stamped by the app at first key store** (keychain, beside the key, write-once per license id, rollback-guarded) and fed to the crate's `evaluate()`. Supersedes "window opens at first failed activation attempt" — same moment in practice (paste always fires an immediate attempt), one less state.
- **Trials ride the same activation rails.** Trial request chains mint + activate in one online exchange, so trial users are Activated instantly; a mid-chain failure falls into the ordinary 7-day Provisional Window. One grace policy (7 days) covers pre-issuance offline, provisional activation, and trial edge cases alike.
- **Update-window plumbing is a boundary translation only.** The `updates` entitlement's day-granular expiry (lapses 00:00 UTC, per the crate) is converted to ms at the adapter and fills the existing `Licensed { update_through_ms, in_window }` wire type — frontend, updater gate, and the fresh-install-after-lapse check (`MNEMA_BUILD_DATE_MS` compile-time build date) all unchanged. Renew button keeps the hardcoded Polar renewal checkout URL. Accepted: windows now end at midnight UTC, not the purchase minute.
- **Key delivery: claim flow + email, both.** Happy path is the new in-app claim: buy button → Polar checkout in browser → success redirect to a `mnema://` deep link (plugin already in-tree from MCP OAuth) → app polls licensegate claim with the checkout id → key installs and activates with no paste. Email delivery continues exactly as today for every purchase — the durable record and the fallback for cross-machine buys, blocked redirects, and re-delivery. The 30-day claim window is acceptable because email, not claim, is the record.
- **Deep-linked key replacement of a healthy license requires confirmation** *(2026-07-18)*. `mnema://license/activate`/claim can be fired by any webpage; if the incoming (signature-valid) key carries a **different license id** than a currently stored key whose status is healthy **Licensed**, the app shows a blocking confirm dialog before overwriting — closing the silent-swap shape (attacker burns a real purchase, replaces the victim's key, refunds it → victim drops to Revoked weeks later with no visible cause). Every other path stays frictionless: no stored key (fresh claim), same license id (replayed link), any unhealthy state, and the Settings paste path (already deliberate).
- **One licensegate deployment, two products.** `mnema` (prod) and `mnema-dev` (Polar sandbox) on one Railway deployment — per-product keypairs/kid/public token/CRL give the isolation; a second deployment would double ops cost to defend against nothing (licensegate is developed locally). Release builds bake the prod pubkey+kid, public token, prod base URL, and a **CRL floor fetched live from the prod CRL endpoint during the CI release build**; dev builds bake the sandbox equivalents via the existing compile-time overrides (`MNEMA_DEV_ACTIVATION_URL` generalizes to one licensegate base-URL override). All dev bypass knobs carry over unchanged; new/renamed `MNEMA_*` vars must enter `turbo.json` `passThroughEnv`.
- **Hard cut, no parallel period.** Confirmed 2026-07-16: zero keys exist in the wild (no orders, no comps). One cycle: deploy licensegate + products, repoint the Polar webhook, ship the new build, delete `services/fulfillment` in the same change. Old-format wire tests die with it, replaced by licensegate's conformance vectors. Old builds in the field lose nothing (offline-verified; CRL staleness never locks).
- **Over-cap/reset is in-app; devices surface as a count, never a list.** The old web reset form dies with the worker. The over-cap screen gains one button — "Free up my devices" → key-authed `POST /v1/reset` → auto-retry activation (cooldown case shows the server's retry date). Settings/licensing UI shows the device *count* from `POST /v1/validate` (`devices: {used, cap}`) — licensegate never lists machines back. *(Amended 2026-07-17, ADR 0055:)* activation now sends a **generic hardware model label** ("MacBook Air (M2, 2022)") — support-side disambiguation in the seller dashboard only; never the personal computer name, which would carry the customer's real name. Config URLs as fallbacks: `buy_url` → Polar checkout, `reset_url` → a support note on mnema.day. Deliberately skipped: per-device list/removal and per-machine deactivate UI (revisit only on real support demand — a named list would amend the privacy promise).
- **Ops posture accepted (2026-07-16), two conditions.** (1) **Custom domain before the first release build** — builds bake the base URL forever, so it must be a domain Mnema owns (e.g. `license.mnema.day`), never a Railway subdomain. (2) **Nightly pg_dump→R2 is the stated disaster plan** — worst case is restoring yesterday's dump and hand-re-minting a day's purchases from Polar (Polar stays the money truth). Single-instance assumptions (rate limiter, sweeper) accepted at this scale. No alerting stack (licensegate v1 by design): monitoring is manual — Polar's per-sale/refund emails prompt a dashboard glance; orphan renewals stay manual-resolution. Revisit at sales volume where per-sale glancing stops happening.
- **Old artifacts are abandoned, never parsed.** New-format artifacts (key, receipt, `first_seen_at`, trial issuance) live under **new keychain account names** in the same service; old entries (trial record, old key/receipt) are never read or deleted — an upgraded machine is indistinguishable from a fresh install to the new code. `licensing_state` is reshaped by a **new migration** (0047 is NOT edited in place — field trial users have it applied; checksum mismatch would panic their update), carrying over only `max_timestamp_ever_seen_ms` (the rollback high-water mark), everything else rebuilt from the keychain.
- **No trial carryover at cutover.** The old keychain `trial_record` path is deleted, not migrated. Every machine — mid-trial or already expired into Read-Only Mode — goes through the new server trial at next capture and gets a fresh 30 days. Accepted: one free second trial per pre-cutover machine, once, because there are ~no field users and two parallel trial systems cost more than the gift.

## Accepted enforcement bounds (2026-07-10)

Known, deliberately-unclosed gaps in enforcement. Each was weighed against ADR 0053's anti-overcorrection stance and accepted with a named revisit trigger; none is a forgotten TODO.

- **Provisional lapse is enforced to within ~24h, not to the minute.** The capture gate reads a cached `LicenseStatus`, recomputed at launch, capture start, every activation retry, and the daily CRL tick — so a machine on **Provisional Window** day 7 that records continuously without restarting can capture up to ~24h past the true lapse before the daily tick flips it to **Read-Only Mode**. Accepted because the only beneficiary is someone holding a signature-valid pasted key — almost always a paying customer with flaky connectivity — while the airtight fix adds an absolute deadline field to a wire type hand-mirrored in three places (`capture-types`, `lib/licensing.ts`, fixtures). The **Trial** keeps its stricter live-clock check (`capture_allowed_at`) because the trial is the zero-cost path. Revisit if the daily recompute cadence ever loosens.
- ~~**The Device Cap can be overshot across datacenters.**~~ *(Obsolete 2026-07-16, ADR 0054: this described the Cloudflare-KV slot store in the deleted Fulfillment worker. licensegate holds slot claims in one Postgres database — the cross-datacenter propagation window no longer exists.)*
- **`ActivationHint` assumes a single license** (unkeyed by license id; `ponytail:` comment at the definition). Matters only if multi-license support ever lands.

## Decisions

- [ADR 0044](../adr/0044-monetize-as-one-time-purchase-with-paid-update-window.md) — the business model (one-time purchase + paid Update Window + Trial → Read-Only Mode).
- [ADR 0045](../adr/0045-licenses-verified-offline-ed25519-polar-merchant-of-record-only.md) — offline Ed25519 verification, updater-gated enforcement, and Polar-as-Merchant-of-Record-only Fulfillment.
- [ADR 0052](../adr/0052-refunded-licenses-die-via-a-signed-revocation-list.md) — refunded licenses die via a signed **Revocation List**; staleness never locks (amends 0045's "keys are non-revocable").
- [ADR 0053](../adr/0053-licenses-activate-once-per-machine-via-a-signed-activation-receipt.md) — mandatory one-time **Activation** per machine with an offline-forever **Activation Receipt**, 3-device cap, self-service reset (supersedes 0045's "no activation server"; widens 0052's CRL charter to leaked keys).
- [ADR 0054](../adr/0054-licensing-moves-onto-licensegate.md) — licensing moves onto **licensegate**: the bespoke Fulfillment worker and in-app verify modules are replaced by the licensegate platform + pinned client crate; server-issued trials, claim flow, in-app device reset (supersedes 0053's implementation, preserving its philosophy).
