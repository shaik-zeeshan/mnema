# Mnema Desktop — UX Audit Report

*Surface-by-surface and end-to-end review of system-status visibility, interaction feedback, flow, hierarchy, affordances, consistency, error/empty handling, accessibility, and copy. All findings are code-verified within Mnema's intentional terminal/green, dark-first, sans=human / mono=machine design language.*

---

## 1. Executive summary

Mnema is, screen for screen, one of the more carefully built desktop UIs you will audit. Almost every surface implements the full state matrix — loading skeletons, empty states with CTAs, humanized error cards with Retry, in-flight spinners, disabled reasons, success receipts — and the privacy-critical surfaces (the access-request consent screen, the onboarding permission gating) are genuinely exemplary. The dark terminal aesthetic is applied with real discipline, color is rarely the *only* signal, and the WKWebView quirks are handled deliberately rather than papered over. **No surface-level finding rose to data-loss or fully-silent-failure.**

**The user's headline question — "is the flow easy to understand for a new user?" — answered directly:** Yes, every journey is *completable*, and the wayfinding *inside* each screen is strong. But the friction is real and it lives almost entirely at the **seams between surfaces** — the moments where one task hands off to another. A new user is well-guided within a screen and then dropped at the threshold of the next.

The five biggest themes:

1. **States that ignore live system status.** The single sharpest issue in the whole app: a brand-new user finishes onboarding, clicks "Start recording," lands on the dashboard — and the most prominent thing on screen tells them to "Press Record in the title bar above," a button that no longer exists because it has become Stop/Pause. The empty state never learns `isCapturing`. The same anti-pattern recurs in Insights (engine error rendered as "no data"), Debug (session-gated cards, post-failure "not queried"), and the paused status pill.

2. **Invisible entry points.** Quick Recall — the door to the entire daily-recall journey — has no titlebar affordance and is *omitted from the keyboard-shortcuts help overlay*. The timeline frame→audio relationship has no "play this moment" control. There is no Subject→Chat bridge at all. The richest search refinement (the Filter Picker) has no visible trigger.

3. **Context hand-offs that drop state or dead-end.** Opening a source from an Ask AI answer is a one-way exit that abandons the answer. A failed chat turn clears the typed question with no retry. Subject→Chat (when self-assembled) opens an empty pane with none of the subject context carried, even though the app hands off context cleanly in every *other* direction.

4. **One genuinely risky data path.** In the provider card, the API key requires an explicit "Save key" click while its sibling fields autosave silently. A user who pastes a key and clicks "Back to app" loses it with no warning — the core "connect a provider" task can silently fail.

5. **A long tail of a11y name/role/live-region gaps and same-concept/different-treatment inconsistencies** — the notification bell, the Ask AI streaming live-region flood, keyboard-unreachable source cards, unannounced async errors, two retry button styles, two retry labels, cooling-trend rendered red in one place and neutral in another, and so on. None block a task; together they erode the "learn-once" consistency the rest of the app earns.

**Bottom line:** the per-screen craft is high; the cross-surface seams are where a new user gets confused. Roughly six seam-level fixes would move this from "works if you know where to look" to genuinely easy.

## 2. Systemic themes

These patterns recur across surfaces. Fixing them at the pattern level is higher-leverage than the individual line items.

### Theme A — Empty/loading/error states that don't reflect live system state
A state branch keys off one local condition (e.g. `frames.length===0`) and ignores the actual runtime condition (`isCapturing`, `engineError`, "no active session"), so the user is shown copy that contradicts reality.
- **Dashboard first-run landing** — "Press Record" shown while recording is live (`+page.svelte:7207-7217`).
- **Insights Overview** — engine fetch error renders as "No categorized activity yet." / "No focus signal yet." while the feed shows an error+Retry card (`Overview.svelte:1340/1387`).
- **Debug** — three session-gated cards describe the same no-session condition three ways, one telling the user to press a disabled button (`+page.svelte:1683/1744/2170`); System Probe shows "not queried yet" after an auto-query *failed* (`1795/1828`).
- **App shell** — "Paused" pill can't distinguish user-pause from auto-pause from low-disk (`capture-controls.svelte.ts:96-115`).

### Theme B — Invisible entry points / undiscoverable affordances
The richest or most important path has no persistent, mouse-discoverable trigger.
- **Quick Recall launcher** — no titlebar affordance, omitted from shortcuts overlay (`+layout.svelte:478-514`).
- **Frame→audio** — no "play this moment" control or shortcut on the stage.
- **Subject→Chat** — no bridge button anywhere in Subjects/SubjectDetail.
- **Filter Picker** — opens only via ⌘F / `/`, advertised nowhere (`quick-recall/+page.svelte:3710`).
- **Calendar trigger** reads as a passive timestamp, not a date picker (`+page.svelte:6679-6683`).
- **Idle source toggles** — 12px unlabeled icons, "off" state near-invisible (`+layout.svelte:1009`).

### Theme C — Context hand-offs that drop state or are one-way
The app builds context hand-offs well in some directions and not others, so the gaps feel like dead-ends.
- **Ask AI source open** abandons the answer; re-summon lands on an empty search (`quick-recall/+page.svelte:312-329, 3576-3587`).
- **Chat turn error** clears the composer with no retry (`Chat.svelte:505, 1056-1059`).
- **Subject→Chat** opens an empty thread, no subject context.
- **Ask AI Stop** forfeits follow-up and "Continue in Chat" (`quick-recall/+page.svelte:1876`).

### Theme D — Inconsistent autosave / async-action feedback
The same autosave engine confirms the edit at the control on some panels and only via a remote footer dot on others.
- **Autosave cue missing** on Privacy, Transcription, Ask AI, Providers rows; present on Capture/Audio/Video/Storage/Appearance (`Privacy.svelte:18`, etc.).
- **Save directory** reads as applied but defers to next launch with no notice (`Storage.svelte:48`).
- **Provider card** mixes click-to-save key with silently-autosaving siblings (`Providers.svelte:122`).

### Theme E — Accessibility: name/role/value & live regions
Reachability and announcement gaps, mostly on async outcomes and dynamic content.
- **Notification bell** — count/severity badge is `aria-hidden`, no live region (`+layout.svelte:1109`).
- **Ask AI streaming** — entire transcript is one `aria-live=polite` region → token-by-token flood (`quick-recall/+page.svelte:4195`).
- **Answer source cards** — `tabindex=-1`, no roving model; audio cards expose zero keyboard actions (`AnswerSourceCard.svelte:79`).
- **Async errors not announced** while successes are (`Providers.svelte:346`, Transcription, AskAi).
- **Filter value list** — dangling `aria-activedescendant` (`quick-recall/+page.svelte:3967`).
- **Settings** — orphan `role="tabpanel"` (`GeneralPanel.svelte:7`), no headings/page title (`SettingsRail.svelte:193`).
- **Debug** — dangling `aria-controls` (`+page.svelte:1278`), unlabeled refresh button (`1839`).
- **Onboarding** — no focus move / announcement on phase transitions (`+page.svelte:108`).
- **Access request** — focus dropped to body in the granted footer (`+page.svelte:487`).

### Theme F — Same concept, different treatment (consistency)
- Retry button **style** differs (timeline ghost vs audio-lane danger-bordered) (`+page.svelte:7609`).
- Retry **label** differs ("Retry" vs "Try again") in Quick Recall (`4232/4083`).
- Cooling/down trend is **muted-neutral** in Subjects index but **danger-red** in SubjectDetail + ConfidenceBar (`SubjectDetail.svelte:878`) — and collides with the "contradicts" stance red.
- "Re-check" vs "Recheck" in one component (`PermissionsBody.svelte:73/129`).
- "CLI Access" vs "local tool" register split (`access/request/+page.svelte:329`).
- Scope label "All retained" vs "All retained history" across the consent popup and Data→Access (`cli-access.svelte.ts:42`).

### Theme G — Destructive actions without confirm/undo
- **Dismiss conclusion** — one click, no confirm/undo/restore (`Overview.svelte:1530`, Subjects, SubjectDetail).
- **Clear key** — deletes keychain credential, no confirm, neutral styling, unlike its `btn--danger` "Remove" sibling (`Providers.svelte:186`).
- **Exclusion chip** — hover-only × re-enables recording of an app with no confirm (`AppPrivacyExclusion.svelte`).

### Theme H — Status color / encoding contradictions
- "Available" model chip painted in **warn (caution) color** while the card goes green (`AskAi.svelte:132`, `Transcription.svelte:221`).
- Paused pill color collapses three distinct conditions (Theme A overlap).

## 4. End-to-end journeys

### Journey 1 — First run: install → onboarding → permissions → first recording → land in app
**Verdict: minor-friction (completable).** Onboarding has genuinely strong wayfinding — a persistent Welcome·Configure·Finish stepper, back paths at every phase, a finale escape hatch ("Just open the dashboard →") that avoids model-download dead-ends, and an actionable "N need attention →" footer jump.
- **[high] Landing contradicts the action just taken.** `finishOnboarding(true)` starts capture then `goto("/")`; the titlebar correctly reads "Recording," but the center stage shows "No frames yet — Press Record in the title bar above," and there is no Record button (it's now Stop/Pause). Self-heals in seconds, but the most prominent message contradicts the user's action at the worst moment.
- **[medium] "Apply recommended defaults" reads as skip-to-done** but still drops the user mid-accordion at the permissions row with no explanation of what changed.
- **[medium] Permissions row hides its blocker state** — on a true first run the ungranted Screen-recording row looks identical to satisfied required rows; the only cue is the far-away footer.
- **[low] "≈ 60 seconds"** undersells a flow with 11 rows, OS round-trips, and optional model downloads.

### Journey 2 — Daily recall: open Quick Recall → search / Ask AI → read answer → open a source frame
**Verdict: minor-friction (works if you know the door).** Once open, the surface is excellent: auto-focus, orientation copy, all states handled, streaming with visible phase labels and a persistent Back path.
- **[high] No in-app way to discover Quick Recall.** Only summon is ⌥Space; the titlebar has no affordance and the shortcuts help overlay *omits* `toggleQuickRecall`. A new user can never find the door.
- **[medium] Opening a source is a one-way exit.** `selectSource` hands off then `closeCurrentWindow()`; the turn is marked seen, so re-summon lands on an empty search — the answer is gone with no warning and no breadcrumb that it survives in Insights→Chat.
- **[low] Successful brokered open is silent** — no "Opened from your answer" confirmation (error path does report).
- **[low] Pristine "Screen · Audio · Ask AI" cues look like buttons but are inert** — a dead signifier on the very first screen.

### Journey 3 — Browse the timeline: pick a day → scrub → read OCR → play its audio + transcript
**Verdict: minor-friction, difficulty back-loaded onto the last step.** Picking a day, scrubbing, and OCR are discoverable enough.
- **[high] No bridge from a frame to "its" audio.** No "play audio for this moment" control or shortcut; the user must recognize tiny color bars in a separate lane and infer the spatial mapping under a faint two-triangle playhead.
- **[high] Listen-while-scrubbing destroys the player.** Any rail scroll closes the drawer (capture-phase handler); wheel-scrubbing emits a scroll event that also closes it; clicking a different bar closes rather than switches. The core "play audio + follow the frames" loop breaks the moment the user scrubs.
- **[medium] Calendar trigger reads as a passive timestamp**, not a date picker (only a small icon + hover tooltip signal it).
- **[low] Calendar jump keeps the picker open** with no landing confirmation and no scrub hint.

### Journey 4 — Explore Insights: Overview → Subjects → detail → ask in Chat
**Verdict: minor-friction, breaks at the literal last step.** The first four legs are comfortable: clear Timeline⇄Insights toggle, sensible Overview default, strong "you are here" rail, two clean back paths.
- **[high] No Subject→Chat bridge.** Zero references to Chat/Ask in SubjectDetail/Subjects; the only outbound action is "view in Timeline →" (which leaves Insights). The user must notice the rail's "New chat" link, open an empty thread, and re-type — with none of the subject context carried, even though the app hands off context cleanly everywhere else.
- **[medium] Detail disclosure is inconsistent** — Overview exhibit cards open true overlay modals (Esc to close); a Subject "detail" is a full drill-in behind a breadcrumb. Two behaviors for one conceptual "see detail."
- **[medium] Chat is absent from the named nav.** The rail lists Overview/Subjects/Context; Chat is reachable only via the "+ New chat" link, which doesn't obviously read as "the AI assistant."

### Journey 5 — Configure & control privacy: Settings → change a provider → manage exclusions → handle a grant
**Verdict: minor-friction.** The Settings entry, labelled rail, exclusion editor, and especially the access-request consent screen are clearly built with strong state handling and a receipt that names "Settings → Data → Access" back.
- **[high] API key can silently fail to save.** "Save key" persists to keychain on click; adjacent Label/Base URL/Endpoint autosave. Paste a key, click "Back to app" without saving → key destroyed on unmount, no warning; the user leaves believing AI is configured.
- **[medium] "Enable AI features" gives no near-control "still not ready" feedback** — the only "AI unavailable" signal is a status card scrolled out of view.
- **[medium] Privacy/AI toggles lack the near-control save cue** present on Storage/Audio/Video — on privacy-critical switches "did it take?" matters.
- **[low] Exclusion chip** — hover-only × re-enables recording with no confirm; overloaded click-to-pause.
- **[low] Scope wording differs** between the consent popup ("All retained") and Data→Access ("All retained history").

## 6. Recommended next steps

**Sequenced action plan — fix the seams first; they carry the most new-user confusion per unit of effort.**

1. **First-run landing (Fix #1).** Make the dashboard empty state `isCapturing`-aware. This is the single most jarring moment in the product and a one-component change.
2. **Quick Recall discoverability (Fix #2).** Add the launcher to the shortcuts overlay and a titlebar affordance. Without this, the entire daily-recall journey is hidden.
3. **The two timeline audio seams (Fixes #3, #4).** Add a "play this moment" bridge and stop the drawer from self-closing on scrub/click. Together they unblock the "listen while watching" loop, the actual point of the timeline surface.
4. **Insights Subject→Chat bridge (Fix #5).** Wire an "Ask AI about {subject}" button through the existing context-handoff bus with a prefilled prompt. The plumbing already exists in every other direction.
5. **API key save model (Fix #6).** Eliminate the silent-loss path — autosave on blur or guard navigation. This is the only place the audit found a core task that can *silently fail*.
6. **Cross-cutting feedback + a11y pass (Theme D + Theme E).** Batch the autosave-cue wiring, the `role="alert"` on async errors, the notification-bell name, and the Ask AI live-region fix — these are mechanical and high-trust.
7. **Consistency sweep (Theme F + Theme H).** Unify retry styles/labels, the cooling-trend color, the "available" chip color, and the small copy seams. Low risk, high polish-per-edit.
8. **Destructive-action guards (Theme G).** Add undo/restore for Dismiss and a confirm/destructive-styling for Clear key.

**Follow-up: a runtime/visual pass is worthwhile.** This audit is code-based and rigorous on logic, markup, and CSS tokens, but several findings hinge on *perceived* timing and contrast — how long the "Press Record" flash actually lasts before the first frame merges, whether the sub-AA placeholders and 9px metadata read as badly in the running WKWebView as the token math predicts, and how disruptive the chat scroll-jacking and audio-drawer auto-close feel in motion. Running the app and walking the five journeys with a stopwatch and a screen reader would confirm the severity calls on the timing- and contrast-sensitive items and likely surface a handful of motion/contrast issues that static review can't see.
