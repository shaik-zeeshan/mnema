◆ Annotations · 02 Day Blocks

What it does

Dayflow's calendar view: the same day rendered as proportional time blocks on an hour grid (8:00–19:00 here), colored by category, with a live "now" line while recording. Beside it, the day's numbers: active time, deep-focus share, category split, top apps, and a capture-density strip that makes gaps honest (away time and un-summarized windows read as empty, not as zero work).

It is the second view of the same Journal sub-tab — a Cards ⇄ Blocks Segmented toggle in the page header, not a separate tab.

How it works

Pure presentation over cached rows — no LLM calls, ever, for this view. Block geometry is layout math over each activity's startedAtMs/endedAtMs; colors come from the existing --cat-* tokens via CATEGORY_COLOR.

  • Blocks: list_user_context_activities for the day window (same call the Cards view makes)
  • Stats rail: get_usage_charts — the command that already powers Overview's top-apps / top-sites / category bar / heatmap exhibits

The now-line is client clock only. Hovering a block shows the Cards view's card as a popover; clicking opens the receipt modal (mockup 04), same as a card click. Hovering a category row in the stats rail dims every other category's blocks on the grid — isolate by hover, no extra control. Layout rules the mockup hardcodes but the build must own: the visible range spans the day's first-to-last activity padded out to the hour (a fixed 8:00–19:00 is only the fallback for empty days); blocks render true-to-duration with only a small visibility floor (~10px), never inflated; and anything under ~10 minutes drops its text entirely — a color sliver that leans on the popover. The shimmer strip between the last block and the now-line is the captured-but-unsummarized region, the same pending treatment as the Journal river's live edge.

Design provenance: Timing (auto-zoom to active hours), Google Calendar (true-proportional short events — don't inflate), Dayflow (text-free slivers under ~10 min), Toggl (make data-confidence visible), Rize (category isolate-on-hover).

Net-new code

  • A Blocks view inside the same new apps/desktop/src/lib/insights/DayTimeline.svelte (or a sibling DayBlocks.svelte if the file nears the 800-line ceiling), switched by the shared Segmented control from lib/settings
  • Optional nicety: one thin "day bundle" Tauri command in apps/desktop/src-tauri/src/user_context/commands.rs returning activities + digest + charts in a single round-trip — an optimization only; the three existing commands work today

Not built: no new tables, no new worker, no new chart backend — get_usage_charts (registered in apps/desktop/src-tauri/src/lib.rs as usage_charts::get_usage_charts) already computes every number in the stats rail.

Data it uses

  • user_context_activities — block positions, colors, focus tags
  • get_usage_charts — active time, per-app/per-site totals, category percentages, hour-level density (drawn from frame/app-metadata aggregation over capture segments)

Semantics worth labeling in the UI: an empty stretch means "not captured" or "worker hasn't summarized it yet" (the User Context worker trails live capture by up to one 30-minute window), never "idle". The density strip disambiguates: capture coverage comes from segment data, so a lit bin under an empty grid region = footage exists, summary pending — and the bin's hover tooltip says exactly that (shown above), instead of a permanent caption explaining the strip.
