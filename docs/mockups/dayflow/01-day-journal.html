◆ Annotations · 01 Day Journal

What it does

A chronological river of activity cards — Dayflow's core view — as a new Journal sub-tab inside Insights. Each card is one AI-written activity (title, 1–2 sentence summary, category color, focus level, apps touched), stacked along a time spine. The day's digest headline + narrative leads the page as the lede, exactly as it does on Overview today.

A card has one action: clicking anywhere on it opens its receipt — the scrubbable real footage behind it (mockup 04). "Open in Timeline" lives inside that modal, so there's no ambiguity about what a click does. The live edge of the river ends in an explicit "summarizing…" slot rather than just stopping, and empty days get designed empty states (below). Category color rides a left edge bar on an otherwise neutral card (plus the chip), so the day scans by color without the cards shouting.

Design provenance (from Dayflow's shipped timeline source): category as an edge bar on a neutral body; the unprocessed tail as a sized, labeled object — its paused variant should double as the Resume button; staggered ~30ms card entrance on load; when a paused/stopped recorder means no new cards, the pending slot says so and is the affordance to fix it.

How it works

Nothing new runs. The User Context worker (apps/desktop/src-tauri/src/user_context/worker.rs, run_forward_activity_window) already LLM-summarizes each ~30-minute capture window in the background while you record — so by the time you open Journal, the cards exist as rows. Rendering this page makes zero LLM calls; it is four reads (grilled Jul 3):

  • list_user_context_activities (range path, now evidence-hydrated) for the day's cards
  • get_user_context_digest for the lede (cached in user_context_digests behind a per-range freshness floor; ↻ Re-read reuses the existing regenerate flow); lede stats come from Overview's computation extracted into a shared helper — same label ⇒ same computation ("tracked" = usage-charts basis, category/focus = activity basis)
  • list_frame_summaries_in_range once for the whole day, bucketed frontend-side — powers the away-gaps, the pending region's "frames exist past the watermark" check, per-card frame counts (= frames still on disk now; 0 ⇒ "footage expired"), and Blocks' density strip later
  • get_user_context_status with the new coveredUntilMs field (the worker's summarized-up-to watermark, read from latest_derivation_run_window())

Gaps render as gaps — a window with no capture reads as away time. The trailing edge is different: frames newer than coveredUntilMs render as the pending slot — "Summarizing this window…" while the engine is healthy, the reason when it is not (engine off / no key / budget exhausted, from the same status read) — so the river never ends in silence a user could mistake for lost data, recording or not, wedged worker or not. Day boundary is frontend local midnight, same as the digest cache key — deliberately not Dayflow's 4am logical day.

Net-new code

Frontend:

  • Extend the InsightsTab union in apps/desktop/src/routes/insights/+page.svelte
  • Add the rail entry in apps/desktop/src/lib/insights/InsightsRail.svelte
  • New apps/desktop/src/lib/insights/DayTimeline.svelte — reuses CATEGORY_COLOR, humanizeMs, windowFor from activity-helpers.ts; extract Overview's lede-stat computation into a shared helper (don't copy it)
  • Timeline jump wiring: get_timeline_window_around_frame (already registered in lib.rs) — called from "Open in Timeline" inside the receipt modal, not from the card itself

Three field-level backend touches (grill outcome, Jul 3):

  • coveredUntilMs added to get_user_context_status (reads the existing latest_derivation_run_window() watermark)
  • Evidence hydration on the range path of list_user_context_activities (recency path already hydrates)
  • is_headline selected into ActivityEvidenceRef + its hand-mirrored TS twin (the column exists and is written; the read path dropped it, leaving the headline encoded only as evidence[0] ordering — which the receipt's time-sorted ticks would destroy)

Not built: no new tables, no new migrations, no new Tauri commands, no new worker, no new LLM prompts. The backend for this entire view shipped with User Context.

Data it uses

  • user_context_activities — title, summary, category (fixed 8-mode taxonomy with existing --cat-* CSS vars), focus (deep/mixed/distracted — the UI deliberately relabels distracted as "Scattered"), startedAtMs/endedAtMs
  • user_context_activity_evidence — frame refs per activity; the is_headline frame anchors the Timeline jump and the receipt poster
  • user_context_digests — the lede's headline + 2–4 sentence narrative

Retention matters here: per ADR 0029, Retention Cleanup never cascades to user_context_* tables — so the journal stays readable long after raw frames age out. An old card just loses its receipt ("footage expired"), not its story. Delete Recent Capture, by contrast, removes overlapping segments' frames but leaves these summaries in place.
